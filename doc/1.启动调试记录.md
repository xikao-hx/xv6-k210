# xv6 + OpenSBI 在 QEMU 上启动的调试记录

## 总体目标

qemu 通过 opensbi 引导启动 xv6。

**目录说明**：
- opensbi 的目录对应 opensbi-0.9
- xv6 对应的目录是 os
- opensbi 的设备树是 dts/xv6-k210.dtb
- build.sh 是编译脚本
- run.sh 是启动 qemu 的脚本

**工具链**：
```bash
export ARCH=riscv
export CROSS_COMPILE=riscv64-unknown-linux-gnu-
export PATH=$PATH:/home/xikao/quard_star_tutorial/toolchain/gcc-riscv64-unknown-linux-gnu/bin/
```

## 任务2：在 qemu 中成功启动 xv6

### 问题1：FW_JUMP_ADDR 与 xv6 内核入口地址不匹配

**现象**：OpenSBI 启动后打印 banner，但之后无任何 xv6 输出。

**调试过程**：

第一步，确认 xv6 内核的入口地址：
```bash
readelf -h output/os/kernel | grep Entry
# 输出: Entry point address: 0x80200000
```

第二步，确认 OpenSBI 跳转地址：
```bash
grep FW_JUMP_ADDR build.sh
# 输出: FW_JUMP_ADDR=0x80020000
```

对比发现 `0x80020000` 和 `0x80200000` 差了 `0x1E000`。OpenSBI 跳转到错误地址，xv6 从未被执行。

同时交叉验证内核链接脚本 `os/kernel/kernel.ld`：
```bash
grep "\. = 0x" os/kernel/kernel.ld
# 输出: . = 0x80200000
```

确认链接地址也是 `0x80200000`，与 readelf 一致。

**修复**：`build.sh` 中将 `FW_JUMP_ADDR` 从 `0x80020000` 改为 `0x80200000`。

---

### 问题2：cpuid() 读 tp 寄存器但无人设置

**现象**：修复 FW_JUMP_ADDR 后，OpenSBI 仍然只输出 banner，无 xv6 输出。

**调试过程**：

第一步，添加最简调试确认 xv6 是否被执行。在 `entry.S` 的 `_entry` 最开始添加写 UART 的代码无输出，怀疑 xv6 根本没跑到，或者跑了但卡死在某处。

第二步，用 GDB 连接 QEMU 断点调试：
```bash
# 终端1: 启动 QEMU 等待 GDB 连接
qemu-system-riscv64 -machine virt -m 128M -smp 1 \
    -bios output/opensbi/fw_jump.bin \
    -kernel output/os/kernel \
    -S -gdb tcp::1234 -nographic

# 终端2: 连接 GDB
riscv64-unknown-linux-gnu-gdb output/os/kernel
(gdb) target remote :1234
(gdb) b _entry
(gdb) c
```

GDB 断在 `_entry`，说明 xv6 确实被执行了。单步跟踪到 `main()` 函数。

第三步，在 `main()` 中 hart 0 应该走 `if(cpuid()==0)` 分支，但实际走了 `else` 分支。在 GDB 中检查 `tp` 寄存器：
```
(gdb) info registers tp
# tp = 0x8001xxxx（一个大地址）
```

这不是 hart ID (0)！追溯到 `cpuid()` 函数（`proc.c:51-54`）：
```c
int cpuid()
{
  int id = r_tp();  // 读 tp 寄存器
  return id;
}
```

第四步，追查 `tp` 应该在哪设置。OpenSBI 通过 `a0` 寄存器传递 hart ID 给下一级。查看 `entry.S`：

```asm
_entry:
    add t0, a0, 1      # a0 只用来算栈偏移
    slli t0, t0, 14
    la sp, boot_stack
    add sp, sp, t0
    call main           # a0 丢失！没有存入 tp
```

`a0` 被用来计算栈偏移，但从未写入 `tp`。`call main` 后 `a0` 可能被 `main` 的 prologue 覆盖。cpuid() 读到的是 `tp` 中的残留值（OpenSBI 设置的 mscratch 指针）。

**修复**：`_entry` 开头添加 `mv tp, a0` 在栈计算之前保存 hart ID。

**验证**：GDB 重新断点，确认 `tp` = 实际 hart ID，`main()` 正确进入 if 分支。

### 问题3：VIRTIO0 地址与 QEMU 实际 MMIO 布局不匹配

**现象**：xv6 打印 "xv6 kernel is booting" 后立即崩溃：

```
scause 0x0000000000000005    # Load access fault
sepc=0x00000000802074ae      # 崩溃地址
stval=0x0000000010100000     # 访问的目标地址
panic: kerneltrap
```

**调试过程**：

第一步，解析崩溃信息：
- `scause=5`：Store/AMO access fault（或 load access fault）
- `stval=0x10100000`：这条是访问了不存在的 MMIO 地址
- `sepc=0x802074ae`：崩溃位置

第二步，用 `objdump` 定位 sepc 对应的源代码行：
```bash
riscv64-unknown-linux-gnu-objdump -S output/os/kernel | grep -B5 "802074ae"
```

发现该地址在 `virtio_disk_init()` 函数内。

第三步，对比 xv6 的 MMIO 定义和 QEMU 实际布局。查看 `memlayout.h`：
```c
#define VIRTIO0 0x10100000
```

第四步，查看 QEMU 源码确认 virtio MMIO 基址：
```bash
grep -n "VIRT_VIRTIO" /home/xikao/qemu-4.2.0/hw/riscv/virt.c
```

```c
[VIRT_VIRTIO] = { 0x10001000, 0x1000 }
```

QEMU 的 virtio MMIO 从 `0x10001000` 开始，而 xv6 尝试访问 `0x10100000`。地址不匹配导致 load access fault。

**修复**：`memlayout.h` 中 `VIRTIO0` 改为 `0x10001000`。

### 问题4：缺少 OpenSBI 下的定时器中断支持

**现象**：修复 VIRTIO0 后，xv6 打印 "xv6 kernel is booting" 和 "init: starting sh"，但 shell 提示符 `$` 不出现，无法交互。

**调试过程**：

第一步，确认用户态 init 进程是否被创建并调度。在 `main()` 最后 `scheduler()` 前后添加 printf，发现 scheduler 运行了但 init 进程似乎没有被调度。

第二步，分析 xv6 进程调度依赖。xv6 的 scheduler 依赖于定时器中断来做时间片轮转。没有定时器中断，init 进程获得 CPU 后如果阻塞在 UART 读上就不再被调度。

第三步，检查 xv6 原 M-mode 的定时器机制。原来在 M-mode 下，`kernelvec.S` 中的 `timervec` 直接操作 CLINT 寄存器设置定时器。改为 OpenSBI + S-mode 后，S-mode 无权访问 CLINT，必须通过 SBI ecall。

第四步，检查当前中断使能状态。阅读 `main.c` 和 `trap.c`，发现三个缺口：

a. `main.c` 只使能了 SEIE（外部中断），未使能 STIE（定时器中断）：
```c
asm volatile("csrs sie, %0" : : "r"(1 << 9));  // SEIE
// 缺少: csrs sie, (1 << 5)  // STIE
```

b. `trap.c` 的 `devintr()` 未处理 `scause=5`（S-mode 定时器中断）：
```c
// 只处理 scause=9（外部中断）和 scause=1（原 M-mode 软件中断）
// 缺少 scause=5（S-mode 定时器中断）
```

验证方式：查阅 RISC-V privileged spec，确认 S-mode 定时器中断的 scause 值为 `0x8000000000000005`（interrupt bit | 5）。

c. `clockintr()` 未重新设置下次定时器，且首次定时器从未被设置：
```bash
grep -n "sbi_set_timer" os/kernel/trap.c os/kernel/main.c
# 修复前无任何结果
```

**修复**三处：
1. `main.c`：添加 `csrs sie, 1<<5` 和首次 `sbi_set_timer()` 调用
2. `trap.c`：`devintr()` 添加 scause=5 分支；`clockintr()` 末尾调用 `sbi_set_timer()`

### 问题5：kinit() 中卡死（PHYSTOP 超过 QEMU 实际内存）

**现象**：添加 scause=5 处理后，xv6 打印 "xv6 kernel is booting" 后无输出，QEMU 超时退出。

**调试过程**：

第一步，缩小范围。在 `main()` 的各个初始化函数前后插入 printf 桩：
```c
printf("before kinit\n");
kinit();
printf("after kinit\n");     // 这行从未打印！
```

确认真实卡在 `kinit()` 内部。

第二步，在 `kinit()` 内部进一步打桩：
```c
void kinit() {
  printf("kinit: before freerange\n");
  freerange(end, (void*)PHYSTOP);
  printf("kinit: after freerange\n");  // 这行从未打印
}
```

确认真实卡在 `freerange()` 循环内。

第三步，在 `freerange()` 和 `kfree()` 内添加计数器：
```c
void freerange(void *pa_start, void *pa_end) {
  char *p;
  int count = 0;
  for (p = (char*)PGROUNDUP((uint64)pa_start); p + PGSIZE <= (char*)pa_end; p += PGSIZE) {
    count++;
    if (count % 1000 == 0)
      printf("freerange: %d pages freed\n", count);
    kfree(p);
  }
}
```

发现循环在执行但始终走不到头，计算总页数后发现 PHYSTOP 定义的物理内存范围远超实际可用内存。

第四步，确认实际可用内存：
```bash
# 查看 DTS 中定义的内存大小
grep -A2 "memory@" dts/xv6-k210.dts
```

输出：
```
memory@80000000 {
    reg = <0x00000000 0x80000000 0x00000000 0x00800000>;
};
```

地址 `0x80000000`，大小 `0x00800000` = 8MB。内核从 `0x80200000` 开始，可用内存最多到 `0x80800000`。

但 `memlayout.h` 中定义：
```c
#define KERNBASE 0x80000000L
#define PHYSTOP (KERNBASE + 128*1024*1024)  // 128MB!
```

PHYSTOP = `0x88000000`，远超 DTS 定义的 `0x80800000`。freerange 循环在访问不存在的物理地址时触发 page fault，此时 trap 框架尚未完整就绪（plic 还没初始化），导致死机。

**修复**：`memlayout.h` 中 `KERNBASE` 改为 `0x80200000`，`PHYSTOP` 改为 `0x80600000`（给内核留 2MB 空间，剩余约 6MB 可用内存）。

## 任务3：在 qemu 中成功启动多核 xv6

### 问题6：多核启动不一致，辅核从不执行

**现象**：
- `-smp 2` 且 boot hart=0 时：xv6 启动成功，shell prompt 出现，但 "hart 1 starting" 从未打印
- `-smp 2` 且 boot hart=1 时：OpenSBI 输出后完全无 xv6 输出

需要反复运行多次才能观察到两种 case：
```bash
# 多次运行观察 OpenSBI 的 Boot HART ID 变化
for i in $(seq 1 10); do
  timeout 5 bash run.sh nographic 2>&1 | grep "Boot HART ID"
done
```

**调试过程**：

第一步，确认问题。通过比对 OpenSBI 输出的 "Boot HART ID" 和 xv6 输出，观察到：
- Boot HART ID=0：有 xv6 输出
- Boot HART ID=1：无 xv6 输出

第二步，理解 OpenSBI 的多核启动流程。仔细阅读 OpenSBI 源码：

```bash
# 追踪关键文件
ls opensbi-0.9/lib/sbi/sbi_init.c    # sbi_init(), init_coldboot(), init_warmboot()
ls opensbi-0.9/lib/sbi/sbi_hsm.c     # HSM hart state machine
ls opensbi-0.9/lib/sbi/sbi_domain.c  # domain finalize
ls opensbi-0.9/firmware/fw_base.S    # low-level boot flow (_start_warm)
```

第三步，画调用流程图。在纸上追踪代码路径：

`sbi_init()` 用 `atomic_xchg(&coldboot_lottery, 1)` 选冷启动 hart：
- 胜者：`init_coldboot()` → 初始化一切 → `sbi_domain_finalize()` → 只启动 domain 的 boot hart
- 败者：`init_warmboot()` → `wait_for_coldboot()` → `sbi_hsm_init(FALSE)` → `sbi_hsm_hart_wait()`

在 `sbi_hsm_hart_wait()` 中：
```c
while (atomic_read(&hdata->state) != SBI_HART_STARTING) {
    wfi();
};
```

辅核在此无限等待，需要有人调用 `sbi_hsm_hart_start()` 将状态从 STOPPED 改为 STARTING。

第四步，在 `sbi_domain_finalize()` 中找到关键逻辑：

```c
sbi_domain_for_each(i, dom) {
    dhart = dom->boot_hartid;     // 根 domain 的 boot_hartid = 冷启动 hart
    if (dhart == cold_hartid) {
        // 只设置冷启动 hart 的下阶段跳转地址
        scratch->next_addr = dom->next_addr;
    } else {
        // 这个 else 分支从未被执行！
        // 因为 dhart 永远等于 cold_hartid（只有一个根 domain）
        rc = sbi_hsm_hart_start(...);
    }
}
```

确认根因：OpenSBI 只启动每个 domain 的一个 boot hart。单 domain 场景下只有冷启动 hart 被启动，其他 hart 永久卡在 HSM wait。

第五步，设计修复方案。有两种选择：
a. 修改 DTS 添加 domain 配置（复杂，需要理解 domain 语义）
b. 在 xv6 中通过 SBI HSM ecall 手动启动辅核（标准做法）

选择方案 b。

第六步，实现 SBI HSM ecall。查阅 SBI v0.2 规范，HSM extension ID = `0x48534D`，`hart_start` function ID = 0。SBI v0.2 使用扩展调用约定：
- `a7` = 扩展 ID
- `a6` = 功能 ID
- `a0-a2` = 参数

在 `sbi.h` 添加 `sbi_hart_start()`。

第七步，修改 main.c 的启动逻辑。原来 `if(cpuid() == 0)` 只在 hart 0 做初始化，改为 `__sync_bool_compare_and_swap` 让第一个到达的 hart 成为 boot leader。boot leader 调用 `sbi_hart_start(0x80200000, hartid)` 逐个启动辅核。

第八步，验证。多次运行确认两种冷启动场景均正常工作：

```bash
for i in $(seq 1 10); do
  timeout 5 bash run.sh nographic 2>&1 | \
    grep -E "(Boot HART ID|hart.*starting)"
done
```

输出涵盖两种场景：
```
Boot HART ID  : 0 → hart 1 starting
Boot HART ID  : 1 → hart 0 starting
```

## 补充问题：QEMU `-m 8M` 无法启动

### 问题7：`-m 8M` 时 OpenSBI 无任何输出，但 DTS 只定义了 8MB 内存

**现象**：
- `run.sh` 中 `-m 128M` 时一切正常
- 改为 `-m 8M` 后，OpenSBI 连 banner 都不打印，QEMU 超时退出
- 但 DTS 中只声明了 8MB 内存、PHYSTOP 也限制在 6MB——从 xv6 视角确实只用了 8MB
- 另一个 xv6-k210 项目使用 RustSBI 配合 `-m 8M` 却可以正常启动

**核心疑问**：`-m`、DTS、PHYSTOP 三者都涉及内存，为什么 DTS 写 8M 可行，但 QEMU `-m` 给 8M 就不行？

**调试过程**：

第一步，确认 `-m`、DTS、PHYSTOP 三者各自的作用范围。在代码中搜索三者的使用位置：

| 参数 | 生效位置 | 作用 |
|------|----------|------|
| `-m 8M` | QEMU 命令行 → `machine->ram_size` → `memory_region_init_ram()` | **QEMU 模拟的物理 RAM 总量**，从 `0x80000000` 开始 |
| DTS `memory@80000000` | 嵌入 fw_jump.bin → OpenSBI → xv6 读取 | **告知操作系统可用的内存范围** |
| `PHYSTOP` | `memlayout.h` → `kinit()` → `freerange()` | **xv6 内核实际使用多少物理页**，从 `KERNBASE` 到 `PHYSTOP` |

三者的关系是：`-m` ≥ DTS memory ≥ PHYSTOP 范围。DTS 告诉 OS "你可以用这么多"，PHYSTOP 告诉内核 "你只用到这"，而 `-m` 保证底层 QEMU 有足够的物理 RAM 支撑这一切。

第二步，逐步缩小 `-m` 阈值，找到 OpenSBI 能启动的最小值：

```bash
for m in 8M 16M 32M 40M 48M 56M 60M 64M 128M; do
    echo -n "-m $m: "
    timeout 3 qemu-system-riscv64 -machine virt -m $m \
        -bios output/opensbi/fw_jump.bin -nographic 2>&1 | \
        grep -c "OpenSBI"
done
```

结果：
```
-m 8M:  0    # 无输出
-m 16M: 0
-m 32M: 0
-m 40M: 0
-m 48M: 0
-m 56M: 0
-m 60M: 0
-m 64M: 1    # 成功打印 OpenSBI banner
-m 128M: 1
```

阈值为 64MB——低于此值 OpenSBI 无法启动。

第三步，用 GDB 定位 OpenSBI 卡住的位置：

```bash
# 终端1：启动 QEMU 并开启 GDB server
qemu-system-riscv64 -machine virt -m 8M \
    -bios output/opensbi/fw_jump.bin -nographic -s &
# 终端2：3 秒后连接 GDB 打断执行
riscv64-unknown-linux-gnu-gdb -q -nx
(gdb) target remote :1234
(gdb) interrupt
(gdb) info registers pc
# pc = 0x8000051c
(gdb) x/4i $pc
# => 0x8000051c:  j    0x80000518
#    0x80000518:  wfi
```

PC 停在 `_start_hang`——OpenSBI 的无限 WFI 死循环。说明 OpenSBI 在早期初始化阶段触发了 fatal error。

第四步，对比 xv6-k210 项目的 RustSBI，确认它能用 `-m 8M` 启动：

```bash
timeout 5 qemu-system-riscv64 -machine virt -m 8M \
    -bios xv6-k210/bootloader/SBI/sbi-qemu \
    -kernel xv6-k210/target/kernel -nographic
```

输出：
```
[rustsbi] RustSBI version 0.1.1
[rustsbi] Platform: QEMU (Version 0.1.0)
...
```

RustSBI 可以正常初始化并跳转到内核。

第五步，对比两个 SBI 实现的关键差异：

```bash
# 检查 RustSBI 的 ELF 结构
readelf -l xv6-k210/bootloader/SBI/sbi-qemu

# 输出：
# LOAD  0x80000000  MemSiz=0x193e8 (~103KB)  # 整个固件仅一个段，占 ~103KB
```

```bash
# 检查 OpenSBI 的 scratch/stack 分配逻辑
grep -n "SBI_SCRATCH_SIZE\|HART_STACK_SIZE" opensbi-0.9/include/sbi/*.h
# SBI_SCRATCH_SIZE = 0x1000 (4KB)
# SBI_PLATFORM_DEFAULT_HART_STACK_SIZE = 8192 (8KB)
```

计算 OpenSBI 内部额外分配（2 个 hart）：
- `_fw_end` ≈ `0x8001C000`（固件 112KB）
- scratch + stack 空间：`2 × (0x1000 + 0x2000)` ≈ 24KB 额外开销
- 总占用约 136KB，在 8MB 内完全放得下

确认不是单纯的"内存不够"，而是更底层的问题。

第六步，阅读 OpenSBI 平台初始化代码，追踪 `_start_hang` 的触发路径：

```c
// sbi_init.c
void sbi_init(struct sbi_scratch *scratch) {
    if ((SBI_HARTMASK_MAX_BITS <= hartid) ||
        sbi_platform_hart_invalid(plat, hartid))
        sbi_hart_hang();  // → _start_hang
    ...
}
```

`fdt_reset_init()` 在 early_init 阶段解析设备树中的 reset 控制器。如果 FDT 解析过程中因内存限制出现异常，可能导致后续 hart 校验失败或 scratch 空间访问越界。

**根因**：

不是内存容量不够，而是 **OpenSBI 的 generic 平台 + `fw_jump` 模式对极小 RAM（< 64MB）存在兼容性问题**。具体原因：

1. **OpenSBI 内部初始化开销**：OpenSBI 在早期启动时需要为所有 hart 建立 scratch 空间、解析 FDT、初始化 domain/HSM/CLINT/PLIC 等子系统。这些操作涉及大量 FDT 解析和内存分配，在 RAM 不足时可能触发未定义行为。

2. **QEMU 内部布局**：QEMU 在 RAM 末尾放置生成的 DTB（即使被 OpenSBI 内嵌 DTB 覆盖，QEMU 仍会预留空间）。`-m 8M` 时 RAM 只有 8MB，QEMU 的内部布局和 OpenSBI 的内部布局可能发生踩踏。

3. **RustSBI 为什么可以**：RustSBI 设计更精简——无 domain 框架、无 HSM 状态机、无 scratch 动态分配。其 ELF 加载方式让内存占用精确可控（~103KB），不像 OpenSBI 需要在固件末尾动态扩展。

**结论与方案**：

保持 `-m 128M`（或至少 64M）。DTS 和 PHYSTOP 独立控制 xv6 可见/可用内存，`-m` 只需保证 QEMU + SBI 有足够物理 RAM 完成启动。三者各司其职，互不冲突：

```
QEMU -m 128M      ← 物理 RAM 总量（固件+内核+空闲）
  └─ DTS: 8MB     ← 告知 xv6 可见范围（0x80000000~0x80800000）
       └─ PHYSTOP ← xv6 实际只用 6MB（0x80200000~0x80600000）
```
