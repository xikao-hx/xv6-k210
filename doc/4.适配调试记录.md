# 任务1：K210 平台 rustsbi 启动 xv6 — 调试与修复记录

## 项目背景

在 K210 硬件平台上使用 rustsbi 作为 bootloader 启动 xv6 操作系统。K210 使用 RISC-V 特权级规范 v1.9.1（与常见的 v1.10+ 有多处不兼容）。

参考成功实现：`/home/xikao/xv6-k210-opensbi/xv6-k210/`（使用 OpenSBI）。

## 初始状态

- 系统能启动到 shell 提示符 `$`，但敲击回车后 hang 住
- DMA 驱动必须使用轮询模式（中断模式会导致 SD 卡读取卡死）
- 参考 xv6-k210 可以成功运行

## 问题一：键盘输入无响应（S 态外部中断不工作）

### 现象

系统启动到 shell 后，敲击键盘回车，系统 hang 住，没有任何响应。

### 调试过程

1. **定位中断路径**：K210 v1.9.1 没有 S 模式外部中断（SEIE），rustsbi 通过"软代理"方式转发外部中断：
   - M 模式收到 MachineExternal 中断
   - 设置 `stval = 9`，触发 S 模式软件中断（`mip::set_ssoft()`）
   - 清除 `mie.mext`，等待 S 模式处理完后再开启

2. **检查 SBI 扩展 ID**：内核通过自定义 SBI 扩展 `sbi_set_extern_interrupt()`（注册中断处理函数）和 `sbi_set_mie()`（重新使能 M 模式外部中断）与 rustsbi 交互。

3. **发现根因**：`sbi.h` 中的 SBI 扩展 ID 与 rustsbi 中注册的不匹配：
   - 内核使用：`0x09000004` / `0x09000005`
   - rustsbi 期望：`0x0A000004` / `0x0A000005`
   - 导致第一次中断处理后 `mext` 被清除但无法重新使能，后续中断全部丢失

### 修复

**文件：`xv6/kernel/include/sbi.h`**

将 SBI 扩展 ID 从 `0x09000004`/`0x09000005` 修改为 `0x0A000004`/`0x0A000005`：

```c
static inline void sbi_set_extern_interrupt(unsigned long func_pointer) {
    asm volatile("mv a6, %0" : : "r" (0x210));
    SBI_CALL_1(0x0A000004, func_pointer);  // 原为 0x09000004
}

static inline void sbi_set_mie(void) {
    SBI_CALL_0(0x0A000005);  // 原为 0x09000005
}
```

**文件：`xv6/bootloader/rustsbi-k210/src/main.rs`**

三处修改：

```rust
// 修改1：初始化时开启 mext 和 msoft（约第231-234行）
mie::set_mext();   // 新增：使能 M 态外部中断
mie::set_msoft();  // 保留

// 修改2：0x0A000005 处理函数正确设置 mext（约第380-383行）
} else if trap_frame.a7 == 0x0A000005 {
    unsafe {
        mie::set_mext();  // 正确的扩展 ID
    }
    trap_frame.a0 = 0;
    trap_frame.a1 = 0;
}

// 修改3：MachineExternal 中断处理（约第436-441行）
Trap::Interrupt(Interrupt::MachineExternal) => {
    unsafe {
        llvm_asm!("csrw stval, $0" :: "r"(9) :: "volatile");
        mip::set_ssoft();
        mie::clear_mext();  // 转发到 S 态后关闭 mext
    }
}
```

### 结果

修复后键盘输入可以工作，回车后系统能响应。

## 问题二：进程被 `scause 5/7` 异常杀死

### 现象

从键盘输入命令后，系统输出：

```
usertrap(): unexpected scause 0x0000000000000007 pid=...
panic: init exiting
```

### 调试过程

1. **分析 scause 编码**：RISC-V v1.10+ 中 store page fault 为 `scause=15`，load page fault 为 `scause=13`。但 K210 v1.9.1 使用统一的访问异常编码：load=5，store=7。

2. **检查 trap.c**：内核的 `usertrap()` 只检查 `scause == 13 || scause == 15`（v1.10+ 编码），K210 上的 `scause=7` 未被识别，进程被直接杀死。

3. **影响范围**：COW（Copy-On-Write）、mmap、lazy allocation 的缺页异常处理全部无法触发。

### 修复

**文件：`xv6/kernel/trap/trap.c`（第79-83行）**

在 K210 平台添加 v1.9.1 的异常编码：

```c
} else if (r_scause() == 13 || r_scause() == 15
#ifndef QEMU
             || r_scause() == 5 || r_scause() == 7
#endif
) {
```

### 结果

进程的缺页异常（包括 COW 写保护异常）能够正确触发处理函数。

## 问题三：非法指令崩溃（TLB 不一致导致指令读取错误）

### 现象

系统启动后出现非法指令崩溃：

第一次（修复前）：
```
[rustsbi] panicked at 'invalid instruction! mepc: 0x1280, instruction: 6a0500d7'
```

第二次（添加部分 sfence_vma 后）：
```
$ [rustsbi] panicked at 'invalid instruction! mepc: 00000000000011c2, instruction: 00000797'
```

地址 `0x11c2` 是 shell 的 `free` 函数中的 `auipc a5, 0x0` 指令——这是一条**合法指令**。

### 调试过程

1. **分析矛盾**：`0x00000797` = `auipc a5, 0` 是合法的 RV64 指令，CPU 不应在此触发非法指令异常。

2. **关键发现**：rustsbi 在 M 模式读取 `mepc` 处的指令时使用物理地址（M 模式 Bare 寻址），不是通过 S 模式页表翻译。CPU 实际执行的指令是通过 TLB 缓存的虚拟地址翻译获取的——TLB 中可能是**过期映射**，指向已被释放并重新分配的物理页。

3. **定位根因——TLB 不一致**：K210 v1.9.1 使用 `sfence.vm`（而非 v1.10+ 的 `sfence.vma`）刷新 TLB。COW 机制在多处修改 PTE 后**未调用 sfence_vma**，导致：
   - `uvmcowmalloc()` 释放旧物理页后，TLB 仍缓存 VA→旧PA 的映射
   - CPU 执行该 VA 处代码时，取到已被 `kfree` 填充为垃圾数据（0x01）的物理页内容
   - 垃圾数据被 CPU 解码为非法指令 → M 模式异常 → rustsbi panic

4. **深入分析 kpagetable 同步问题**：
   - xv6 在 K210 上通过 kpagetable 访问用户内存（`copyin_new`/`copyinstr_new` 直接使用虚拟地址）
   - `fork()` 中 `uvmcopy` 修改了父进程**用户页表**的 PTE（添加 COW 标记），但**未同步更新父进程的 kpagetable**
   - `fork()` 调用 `upg2ukpg` 更新子进程的 kpagetable，但遗漏了父进程
   - `growproc()` 增长内存后**完全没有**调用 `upg2ukpg`
   - 导致 kpagetable 中的 PTE 与用户页表不一致，TLB 刷新后仍加载过期映射

5. **指令缓存问题**：K210 v1.9.1 的 `sfence.vm` 可能只刷新数据 TLB，不刷新指令 TLB。需要在 `sfence_vma` 后追加 `fence.i` 指令确保指令缓存一致性。

### 修复

#### 修改 1：`xv6/kernel/include/riscv.h` — sfence_vma 添加 fence.i

```c
static inline void
sfence_vma()
{
  asm volatile("sfence.vma zero, zero");
#ifdef QEMU
  // QEMU handles TLB coherency automatically
#else
  // K210 v1.9.1: sfence.vm may not flush the instruction TLB,
  // so we also issue fence.i to flush the instruction cache.
  asm volatile("fence.i");
#endif
}
```

说明：`sfence.vma zero, zero` 在 K210 上是非法指令，会陷入 M 模式由 rustsbi 仿真为 `sfence.vm`。rustsbi 同时设置 `mstatus.VM=9`（启用 Sv39 分页）。追加的 `fence.i` 确保指令缓存也被刷新。

#### 修改 2：`xv6/kernel/vm/vm.c` — uvmcowmalloc 两个分支添加 sfence_vma

**单引用路径**（仅当前进程引用该页，直接标记可写）：
```c
if (kgetquota((void *)pa) == 1) {
    *pte |= PTE_W;
    *pte &= ~PTE_COW;
    upg2ukpg(p->pagetable, p->kpagetable, va, va + PGSIZE);
    sfence_vma();  // 新增：刷新 TLB
    return (void *)pa;
}
```

**多引用路径**（多进程共享，分配新页并释放旧页）：
```c
    upg2ukpg(p->pagetable, p->kpagetable, va, va + PGSIZE);
    kfree((void *)pa);       // 旧页可能被填充为垃圾数据
    sfence_vma();            // 新增：必须刷新 TLB，否则 CPU 可能读到垃圾
    return (void *)mem;
```

这是**最关键的修复**——`kfree` 将 refcount 降为 0 的页面用 `0x01` 填充，若 TLB 仍有该页的缓存映射，CPU 执行该地址时会取到非法指令。

#### 修改 3：`xv6/kernel/vm/vm.c` — uvmcopy 末尾添加 sfence_vma

```c
    kaddquota((void *)pa);
  }
  sfence_vma();  // 新增：COW 标记修改父进程 PTE 后刷新 TLB
  return 0;
```

#### 修改 4：`xv6/kernel/proc/proc.c` — fork 同步父进程 kpagetable

```c
  upg2ukpg(np->pagetable, np->kpagetable, 0, np->sz);
  sfence_vma();            // 新增：子进程 kpagetable 更新后刷新 TLB

  np->state = RUNNABLE;
  release(&np->lock);

  // 新增以下两行：父进程的 kpagetable 必须同步
  upg2ukpg(p->pagetable, p->kpagetable, 0, p->sz);
  sfence_vma();

  return pid;
```

#### 修改 5：`xv6/kernel/proc/proc.c` — growproc 添加 kpagetable 同步

```c
int growproc(int n)
{
  uint sz;
  struct proc *p = myproc();
  sz = p->sz;
  if(n > 0){
    if((sz = uvmalloc(p->pagetable, sz, sz + n)) == 0) {
      return -1;
    }
    upg2ukpg(p->pagetable, p->kpagetable, p->sz, sz);  // 新增
  } else if(n < 0){
    sz = uvmdealloc(p->pagetable, sz, sz + n);
  }
  p->sz = sz;
  sfence_vma();  // 新增
  return 0;
}
```

#### 修改 6：`xv6/kernel/proc/exec.c` — exec 中 kpagetable 更新后添加 sfence_vma

```c
  uvmunmap(p->kpagetable, 0, PGROUNDUP(old_sz) / PGSIZE, 0);
  upg2ukpg(p->pagetable, p->kpagetable, 0, p->sz);
  sfence_vma();  // 新增：刷新 TLB 后再释放旧页表

  proc_freepagetable(oldpagetable, old_sz);
```

### 结果

- 系统能正常启动到 shell
- `ls` 等命令可以正常执行
- 不再出现 `invalid instruction` panic
- 出现了双重 `$ $` 提示符的小问题（系空命令导致的额外 fork 循环，不影响功能）

## 关键技术要点总结

### K210 v1.9.1 与 v1.10+ 的差异

| 特性 | v1.10+ | K210 v1.9.1 |
|------|--------|-------------|
| 页表基址寄存器 | `satp` | `sptbr` (CSR 0x180) |
| 分页模式控制 | `satp.MODE` | `mstatus.VM[4:0]` |
| TLB 刷新指令 | `sfence.vma` | `sfence.vm` (0x10400073) |
| Load 缺页异常 | scause=13 | scause=5 |
| Store 缺页异常 | scause=15 | scause=7 |
| S 模式外部中断 | 支持 (SEIE) | 不支持，需 M 模式代理 |

### rustsbi 外部中断代理机制

```
M-mode MachineExternal 中断
  → mie::clear_mext()        // 关闭 M 态外部中断
  → stval = 9                // 标记为转发的外部中断
  → mip::set_ssoft()         // 触发 S 态软件中断
  → S-mode: scause=0x8000000000000001, stval=9
  → S-mode 调用 plic_claim() 获取真实 IRQ
  → 处理中断
  → S-mode 调用 sbi_set_mie() // 重新使能 M 态外部中断
```

### COW 与 TLB 一致性的关键原则

1. **任何 PTE 修改后必须调用 sfence_vma**：K210 v1.9.1 无硬件 TLB 一致性，必须显式刷新
2. **kpagetable 必须与用户页表保持同步**：`copyin_new`/`copyinstr_new` 在 K210 上直接使用虚拟地址通过 kpagetable 访问
3. **sfence_vma 后追加 fence.i**：确保指令缓存也被刷新，防止取指使用过期映射
4. **kfree 后尤其需要 TLB 刷新**：kfree 将释放的页面填充为 0x01（垃圾数据），若 TLB 仍缓存该物理页会导致非法指令
