# 任务1：OpenSBI K210 串口输出调试与 FPIOA 适配

## 目标

使 OpenSBI 在 K210 开发板上正常启动并输出串口信息。

## 背景

K210 芯片使用 FPIOA（Field Programmable IO Array）管理引脚功能复用。上电后所有 IO 处于默认复位状态，必须配置 FPIOA 将 UARTHS 功能路由到物理引脚（IO4=RX, IO5=TX），UART 才能工作。

原 xv6-k210 项目使用 RustSBI 作为 M 态固件，RustSBI 通过 k210-hal 库完成 FPIOA 和 UART 初始化。本项目将其替换为 OpenSBI v0.9，需要在 `platform/kendryte/k210/` 中适配 K210 硬件。

## 调试过程

### 现象

固件烧录成功（kflash.py 输出 `Boot to Flashmode Successfully`、`Programming BIN: 100%`），但串口完全无输出——连 OpenSBI 的 banner 信息都不出现。

### 排查步骤

**1. 对比 RustSBI 初始化流程**

RustSBI（`xv6-k210/bootloader/SBI/rustsbi-k210/src/main.rs`）的初始化顺序：

```
SYSCTL.constrain() → FPIOA.split(&sysctl.apb0)
  → io5.into_function(UARTHS_TX)
  → io4.into_function(UARTHS_RX)
  → Clocks::new()
  → UARTHS.configure(115200.bps, &clocks)
```

我们的 OpenSBI `platform.c` 中 `k210_console_init()` 调用链：

```
k210_fpioa_init() → sifive_uart_init(base, freq, baud)
```

基本等价，但需逐项核对细节。

**2. 检查 FPIOA 寄存器配置值**

对比 K210 官方 SDK（`参考资料/kendryte-standalone-sdk/lib/drivers/fpioa.c`）中 `function_config[]` 表的默认值：

| 引脚 | 字段 | SDK 默认值 | 原代码值 | 原代码 hex |
|------|------|-----------|---------|-----------|
| IO4 (RX) | ch_sel | 18 (UARTHS_RX) | 18 | — |
| IO4 (RX) | ie_en | **1** | **0** ← 错误 | 0x00810012 |
| IO4 (RX) | st | 1 | 1 | — |
| IO4 (RX) | pu | 0 | **1** ← 错误 | — |
| IO5 (TX) | ch_sel | 19 (UARTHS_TX) | 19 | — |
| IO5 (TX) | ds | 0xf | 0xf | — |
| IO5 (TX) | oe_en | 1 | 1 | — |

**3. 发现链接地址错误（根因）**

`objdump -h fw_jump.elf` 显示 `.text` 段起始于 `0x80020000`，但 `config.mk` 中 `FW_TEXT_START=0x80000000`。ROM 将固件加载到 `0x80000000`，绝对地址偏移 128KB 导致启动即崩溃。

原因是 `fw_jump.elf.ld` 链接脚本是旧构建的缓存文件。OpenSBI 构建系统仅在 `.ldS` 源文件变化时重新预处理链接脚本，Make 变量（`FW_TEXT_START`）变化不会触发重新生成。

## 代码修改

### 1. `platform.h` — 新增 FPIOA 寄存器位定义

```c
/* FPIOA IO register bit offsets */
#define FPIOA_CH_SEL_SHIFT  0
#define FPIOA_CH_SEL_MASK   0xFF
#define FPIOA_DS_SHIFT      8
#define FPIOA_DS_MASK       0xF
#define FPIOA_OE_EN         (1 << 12)
#define FPIOA_OE_INV        (1 << 13)
#define FPIOA_DO_SEL        (1 << 14)
#define FPIOA_DO_INV        (1 << 15)
#define FPIOA_PU            (1 << 16)
#define FPIOA_PD            (1 << 17)
#define FPIOA_SL            (1 << 19)
#define FPIOA_IE_EN         (1 << 20)
#define FPIOA_IE_INV        (1 << 21)
#define FPIOA_DI_INV        (1 << 22)
#define FPIOA_ST            (1 << 23)

#define FPIOA_IO_CFG(ch, ds, flags) \
    (((ch) & FPIOA_CH_SEL_MASK) | \
     (((ds) & FPIOA_DS_MASK) << FPIOA_DS_SHIFT) | \
     (flags))
```

### 2. `platform.c` — 重写 `k210_fpioa_init()`

```c
static void k210_fpioa_init(void)
{
    /* Enable APB0 bus clock for FPIOA and UARTHS */
    writel(readl((void *)(K210_SYSCTL_BASE_ADDR + K210_CLK_EN_CENT)) |
           K210_CLK_EN_CENT_APB0,
           (void *)(K210_SYSCTL_BASE_ADDR + K210_CLK_EN_CENT));
    /* Read-back to ensure clock enable takes effect before FPIOA access */
    readl((void *)(K210_SYSCTL_BASE_ADDR + K210_CLK_EN_CENT));

    /* Enable FPIOA peripheral clock */
    writel(readl((void *)(K210_SYSCTL_BASE_ADDR + K210_CLK_EN_PERI)) |
           K210_CLK_EN_PERI_FPIOA,
           (void *)(K210_SYSCTL_BASE_ADDR + K210_CLK_EN_PERI));
    /* Read-back to ensure clock enable takes effect before FPIOA access */
    readl((void *)(K210_SYSCTL_BASE_ADDR + K210_CLK_EN_PERI));

    /*
     * Configure FPIOA pins for UARTHS (values from K210 SDK defaults):
     *   IO4 (UARTHS_RX): input enabled, schmitt trigger
     *   IO5 (UARTHS_TX): max drive strength, output enabled
     */
    writel(FPIOA_IO_CFG(K210_FUNC_UARTHS_RX, 0, FPIOA_IE_EN | FPIOA_ST),
           (void *)(K210_FPIOA_BASE_ADDR + 4 * 4));
    writel(FPIOA_IO_CFG(K210_FUNC_UARTHS_TX, 15, FPIOA_OE_EN),
           (void *)(K210_FPIOA_BASE_ADDR + 5 * 4));
}
```

# 任务2：sfence.vma 指令模拟（RISC-V v1.9.1 兼容）

## 目标

在 OpenSBI M 态捕获 xv6 发出的 `sfence.vma` 指令（v1.11 特权规范），转换为 K210 支持的 `sfence.vm` 指令（v1.9.1 特权规范）。

## 背景

K210 芯片使用 RISC-V 特权规范 v1.9.1，该版本与 v1.11+ 在以下 CSR 和行为上有差异：

| 功能 | v1.11+ | v1.9.1 (K210) |
|------|--------|---------------|
| TLB 刷新指令 | `sfence.vma` (0x12000073) | `sfence.vm` (0x10400073) |
| 页表基址寄存器 | `satp` (CSR 0x180) | `sptbr` (CSR 0x180) |
| 分页使能 | `satp.MODE` 字段 (bits 63:60) | `mstatus.VM` 字段 (bits 28:24) |
| Sv39 模式值 | MODE = 8 | VM = 9 |

xv6 内核按 v1.11 规范编写，使用 `sfence.vma` 刷新 TLB。在 K210 上执行 `sfence.vma` 会触发非法指令异常（CAUSE_ILLEGAL_INSTRUCTION），M 态的 OpenSBI 需要捕获并模拟。

## 模拟流程

```
xv6 执行 sfence.vma
  → 非法指令异常 (mcause=2, mtval=0x12000073)
  → OpenSBI sbi_trap_handler()
  → sbi_illegal_insn_handler()
  → system_opcode_insn()
  → 检测到 sfence.vma 编码
  → csrr satp → 取 PPN → csrw sptbr
  → csrr mstatus → 清 VM → 设 Sv39 → csrw mstatus
  → sfence.vm (0x10400073)
  → mepc += 4
  → mret 返回 xv6
```

## 代码修改

### `opensbi-0.9/lib/sbi/sbi_illegal_insn.c` — `system_opcode_insn()`

在 CSR 模拟之前检测 `sfence.vma` 指令编码（mask 0xFE007FFF，值 0x12000073）：

```c
if ((insn & 0xFE007FFF) == 0x12000073) {
    ulong satp_val = csr_read(CSR_SATP);
    ulong ppn = satp_val & SATP64_PPN;
    ulong mstatus_val;

    // 将 PPN 写入 sptbr（CSR 0x180，仅保留低 38 位）
    csr_write(CSR_SATP, ppn & 0x3FFFFFFFFFUL);

    // 设置 mstatus.VM = 9 (Sv39)
    mstatus_val = csr_read(CSR_MSTATUS);
    mstatus_val &= ~0x1F000000UL;   // 清 bits 28:24
    mstatus_val |= (9UL << 24);     // Sv39
    csr_write(CSR_MSTATUS, mstatus_val);

    // 执行 v1.9.1 的 sfence.vm
    __asm__ __volatile__(".word 0x10400073" ::: "memory");

    // 跳过当前指令
    regs->mepc += 4;
    return 0;
}
```

## 任务3：UART 驱动 SBI 化重写

### 背景

原 xv6 的 UART 驱动在 [uart.c](../os/kernel/driver/uart.c) 中直接 MMIO 操作 ns16550a 寄存器：

```c
// 原 putc
while ((ReadReg(UART0, LSR) & LSR_TX_IDLE) == 0)
    ;
WriteReg(UART0, THR, c);

// 原 getc
if (ReadReg(UART0, LSR) & LSR_RX_READY)
    return ReadReg(UART0, RHR);
```

K210 使用的是 SiFive UART（32-bit 寄存器，不同于 16550a 的 8-bit 寄存器），直接 MMIO 会发送垃圾数据。因此将全部数据收发改为 SBI ecall，由 OpenSBI（M 态）管理硬件。

### 修改内容

| 函数 | 原实现 | 新实现 |
|---|---|---|
| `uartputc_sync()` | 轮询 LSR 后写 THR | `sbi_console_putchar(c)` |
| `uartgetc()` | 读 LSR 后读 RBR | `sbi_console_getchar()` |
| `uartstart()` | 轮询 LSR 后写 THR | `sbi_console_putchar(c)` |
| `uartinit()` | 配置波特率、FIFO、IER | 仅初始化软件状态 |

最终代码见 [uart.c:27-117](../os/kernel/driver/uart.c#L27-L117)。

## 任务4：重复 UART0 映射修复

### 现象

内核启动时 panic：

```
panic: remap
```

### 分析

`remap` panic 发生在 `kvmmap()` 检测到虚拟地址已有有效 PTE 时。原因是 `kvminit()` 中 K210 分支先映射了 `0x38000000-0x38100000` 范围（涵盖 UART0），而代码后面又尝试为 `UART0` 再做一次单独映射。

### 修复

将 `kvminit()` 和 `ukvminit()` 中 QEMU 和 K210 的外设映射改为互斥的 `#ifdef` 结构：

```c
// kvminit() in os/kernel/vm/vm.c
#ifdef QEMU
  kvmmap(UART0, UART0, PGSIZE, PTE_R | PTE_W);
  kvmmap(VIRTIO0, VIRTIO0, PGSIZE, PTE_R | PTE_W);
#else
  // K210: UARTHS+GPIOHS (0x38000000-0x38100000, includes UART0 at 0x38000000)
  kvmmap(0x38000000, 0x38000000, 0x100000, PTE_R | PTE_W);
  // K210: DMAC, GPIO, SPI, FPIOA, SYSCTL (0x50000000-0x50400000)
  kvmmap(0x50000000, 0x50000000, 0x400000, PTE_R | PTE_W);
#endif
// CLINT and PLIC follow (common to both platforms)
```

见 [vm.c:23-57](../os/kernel/vm/vm.c#L23-L57)。

## 任务5：S 态外部中断转发

### 背景

K210 使用 RISC-V 特权规范 v1.9.1，其中 **mideleg[9] (SEIE 委派) 不存在**。这意味着外部中断无法直接委派到 S 态，永远到达 M 态。

QEMU 使用 v1.11+ 规范，mideleg[9] 可用，外部中断直接到达 S 态。

参考 xv6-k210 的实现，OpenSBI 在 M 态捕获外部中断后转发为 **SSI (Supervisor Software Interrupt, cause=1)**，并用 `stval=9` (IRQ_S_EXT) 标记这是转发的外部中断。

### 修改：`opensbi-0.9/lib/sbi/sbi_trap.c`

在 `sbi_trap_handler()` 的 `IRQ_M_EXT` 分支中新增转发逻辑：

```c
case IRQ_M_EXT:
    // 清除 MIE/MPIE 防止中断立即重入
    regs->mstatus &= ~MSTATUS_MIE;
    regs->mstatus &= ~MSTATUS_MPIE;
    
    // 构造 SSI + stval=9 注入 S 态
    trap.epc = regs->mepc;
    trap.cause = IRQ_S_SOFT | (1UL << (__riscv_xlen - 1));
    trap.tval = IRQ_S_EXT;    // stval=9 → forwarded external
    trap.tval2 = 0;
    trap.tinst = 0;
    rc = sbi_trap_redirect(regs, &trap);
    break;
```

见 [sbi_trap.c:235-254](../opensbi-0.9/lib/sbi/sbi_trap.c#L235-L254)。

## 任务6：Vendor Ecall 实现

### 背景

xv6-k210 在中断处理后需要调用 `sbi_set_mie()` 重新使能 M 态外部中断（因为转发时 MIE 被清除）。该函数以及 `sbi_set_extern_interrupt` 是 RustSBI 的 vendor 扩展，OpenSBI 原生不支持。

### 修改：`opensbi-0.9/platform/kendryte/k210/platform.c`

新增 `k210_vendor_ext_provider()` 处理两个 vendor EID：

| EID | 功能 | 实现 |
|---|---|---|
| `0x0A000004` | `sbi_set_extern_interrupt` | no-op，兼容 RustSBI |
| `0x0A000005` | `sbi_set_mie` | `csr_set(CSR_MSTATUS, MSTATUS_MIE)` |

```c
static int k210_vendor_ext_provider(long extid, long funcid, ...)
{
    switch (extid) {
    case 0x0A000004:  return 0;  // no-op for RustSBI compat
    case 0x0A000005:  csr_set(CSR_MSTATUS, MSTATUS_MIE); return 0;
    default:          return SBI_ENOTSUPP;
    }
}
```

见 [platform.c:162-178](../opensbi-0.9/platform/kendryte/k210/platform.c#L162-L178)。

## 任务7：trap.c devintr 双平台适配

### xv6 S 态中断检测

```c
#ifdef QEMU
    // QEMU: SEI (cause=9) 直接到达 S 态
    if ((0x8000000000000000L & scause) && 9 == (scause & 0xff))
#else
    // K210: SSI (cause=1) + stval=9, OpenSBI 转发
    if (0x8000000000000001L == scause && 9 == r_stval())
#endif
```

中断处理完成后，K210 需清除 SIP 并调用 `sbi_set_mie()` 重新开启 M 态中断：

```c
#ifndef QEMU
    w_sip(r_sip() & ~2);   // clear SSIP
    sbi_set_mie();          // re-enable M-mode interrupts
#endif
```

timer 中断 (`scause=0x8000000000000005`) 直接由 OpenSBI 的 STI 转发处理，两个平台一致。

### trapinithart 中断使能

```c
void trapinithart(void) {
    w_stvec((uint64)kernelvec);
    w_sstatus(r_sstatus() | SSTATUS_SIE);
    w_sie(r_sie() | SIE_SEIE | SIE_SSIE | SIE_STIE);
    sbi_set_timer(r_time() + 1000000);
}
```

同时使能 SEIE、SSIE、STIE，覆盖 QEMU（用 SEIE）和 K210（用 SSIE）两套中断路径。

见 [trap.c:30-39](../os/kernel/trap/trap.c#L30-L39) 和 [trap.c:214-263](../os/kernel/trap/trap.c#L214-L263)。

## 任务8：QEMU 无法输入修复

### 现象

QEMU 启动后内核输出正常：

```
OpenSBI v0.9
Platform Name             : Generic
...
xv6 kernel is booting
hart 0 starting
init: starting sh
$
```

但按键盘无任何响应。

### 排查过程

1. **确认 PLIC 使能** — `plicinit()` 设置 UART_IRQ=10 优先级为 1，`plicinithart()` 在 S-mode 使能 UART_IRQ。正确。

2. **确认 S 态中断使能** — `trapinithart()` 中 `w_sie(r_sie() | SIE_SEIE | ...)`。正确。

3. **检查 UART 中断源头** — 发现 [OpenSBI 的 uart8250_init](../opensbi-0.9/lib/utils/serial/uart8250.c#L99-L100) 显式禁用所有中断：
   ```c
   /* Disable all interrupts */
   set_reg(UART_IER_OFFSET, 0x00);
   ```
   SiFive UART 同样： [sifive-uart.c:94-95](../opensbi-0.9/lib/utils/serial/sifive-uart.c#L94-L95)
   ```c
   /* Disable interrupts */
   set_reg(UART_REG_IE, 0);
   ```

4. **中断链断裂点确认**：UART IER=0 → 接收数据不产生中断 → PLIC 不触发 → CPU 不进入 devintr → uartintr 永不调用 → 输入失效。

5. **为什么输出正常**：`sbi_console_putchar()` 是同步阻塞调用，OpenSBI 内部轮询 UART 状态直接发送，不依赖中断。

### 修复

在 [uartinit()](../os/kernel/driver/uart.c#L27-L44) 中一次性 MMIO 写使能接收中断：

```c
void uartinit(void)
{
    uart_tx_w = 0;
    uart_tx_r = 0;
    initlock(&uart_tx_lock, "uart");

#ifdef QEMU
    // ns16550a: IER offset 1, bit 0 = RDA (Received Data Available)
    *(volatile uint8*)(UART0 + 1) = 0x01;
#else
    // SiFive UART: IE register offset 0x10, bit 1 = RXWM (Receive Watermark)
    *(volatile uint32*)(UART0 + 0x10) = 0x02;
#endif
}
```

### 寄存器布局差异

| 平台 | UART 类型 | 基址 | IER/IE 偏移 | 访问宽度 | 使能位 |
|---|---|---|---|---|---|
| QEMU | ns16550a | `0x10000000` | 1 | u8 | RDA (bit 0) |
| K210 | SiFive UART | `0x38000000` | 0x10 | u32 | RXWM (bit 1) |

默认参数来源于 [fdt_helper.c:21-22](../opensbi-0.9/lib/utils/fdt/fdt_helper.c#L21-L22)：
```c
#define DEFAULT_UART_REG_SHIFT    0
#define DEFAULT_UART_REG_IO_WIDTH 1
```

## 系统中断全链路图

```
┌──────────────────────────────────────────────────┐
│                    硬件层                          │
│  UART RX → UART.IER[bit0/1]=1 → 产生中断信号       │
│       → PLIC 路由 → CPU 外部中断引脚               │
└──────────────────┬───────────────────────────────┘
                   │
    ═══════════════╪══════════════════════════════════
    QEMU (v1.11+)  │  K210 (v1.9.1)
    mideleg[9]=1   │  mideleg[9] 不存在
    SEI → S-mode   │  MEI → M-mode
    ═══════════════╪══════════════════════════════════
                   │
    ┌──────────────┴────────────────┐
    │        OpenSBI M-mode          │
    │  sbi_trap_handler()            │
    │    IRQ_M_EXT:                  │
    │      清 MIE/MPIE               │
    │      构造 SSI+stval=9          │
    │      sbi_trap_redirect()       │
    │        → 写 SCAUSE/SEPC/STVAL  │
    │        → MEPC=STVEC            │
    │        → MRET 进入 S-mode      │
    └──────────────┬────────────────┘
                   │
    ┌──────────────┴────────────────┐
    │         xv6 S-mode             │
    │  kernelvec → kerneltrap()     │
    │    → devintr()                │
    │      QEMU: scause=9 (SEI)     │
    │      K210: scause=1+stval=9   │
    │    → plic_claim() → uartintr()│
    │    → plic_complete()          │
    │    → sbi_set_mie() [K210]     │
    └───────────────────────────────┘
```
