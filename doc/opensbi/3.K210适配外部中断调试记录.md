# K210 外部中断转发与 SD 卡 DMA 读取调试记录

## 总体目标

在 K210 实机硬件上，使用 OpenSBI v0.9 作为 M-mode 固件，让 xv6 通过 SPI + DMA 成功读取 SD 卡（FAT32 文件系统初始化）。

**背景**：K210 使用 RISC-V v1.9.1 特权级规范，该版本中 `mideleg[9]`（外部中断委托位）不存在——外部中断无法委托给 S-mode，始终在 M-mode 处理。OpenSBI 需要手动将 M-mode 外部中断转发给 S-mode。

**工具链**：
```bash
export ARCH=riscv
export CROSS_COMPILE=riscv64-unknown-linux-gnu-
export PATH=$PATH:/home/xikao/quard_star_tutorial/toolchain/gcc-riscv64-unknown-linux-gnu/bin/
```

## 问题 1：`panic: kerneltrap: interrupts enabled`

### 现象

xv6 启动后在 `fat32_init` → `sdcard_read_sector` → `sd_read_data_dma` 路径中触发 panic：

```
panic: kerneltrap: interrupts enabled
```

### 调试过程

第一步，定位 panic 位置。查看 `os/kernel/trap/trap.c` 的 `kerneltrap()` 函数：

```c
void kerneltrap()
{
  ...
  if(intr_get() != 0)
    panic("kerneltrap: interrupts enabled");
  ...
}
```

`intr_get()` 检查 `sstatus` 寄存器的 `SIE` 位。panic 表示在 S-mode 异常处理入口处，SIE 位被置位（应为 0，硬件应在 trap 入口自动清除 SIE）。

第二步，分析可能原因。SIE 不会被硬件之外的手段设置，唯一可能是：
- OpenSBI 在 M-mode 处理完中断后，通过 `mret` 返回 S-mode 时，`sstatus.SIE` 被错误恢复为 1

第三步，对比 RustSBI 的实现。用户提示参考 `xv6-k210/bootloader/SBI/rustsbi-k210/src/main.rs`：

```rust
// RustSBI timer handler (line 428-433)
fn handle_mtimer() {
    mie::clear_mext();    // 清除 MEIE
    mie::clear_mtimer();  // 清除 MTIE
    // ... 设置 STIP
}

// RustSBI external handler (line 436-441)  
fn handle_mext() {
    mie::clear_mext();    // 清除 MEIE
    mie::clear_mtimer();  // 清除 MTIE
    // ... 设置 SSIP
}
```

第四步，对比 OpenSBI 的 `sbi_timer.c`。OpenSBI 的 `sbi_timer_process()` 原实现只清除了 MTIE，未清除 MEIE：

```c
// 修复前
void sbi_timer_process(void)
{
    csr_clear(CSR_MIE, MIP_MTIP);     // 只清除 MTIE
    csr_set(CSR_MIP, MIP_STIP);
}
```

### 根因

M-mode 中断转发的竞态条件：

1. M-mode 定时器中断触发 → `sbi_timer_process()` 设置 STIP、清除 MTIE（但保留 MEIE）
2. `mret` 返回 S-mode 之前，如果 MEIP 挂起（外部中断刚好到达），MEIE 未清除 → M-mode 外部中断立即触发
3. 外部中断处理中：设置 SSIP、清除 MEIE/MTIE、`mret` 使能 MIE 并返回 S-mode
4. S-mode 入口时，`sstatus.SIE` 被错误恢复为 1

因为 M-mode 外部中断在返回 S-mode 的临界区中打断，破坏了 `sstatus` 的 SIE→SPIE 保存语义。

### 修复（两处）

**修复 1** — `opensbi-0.9/lib/sbi/sbi_timer.c` 的 `sbi_timer_process()`：

```c
void sbi_timer_process(void)
{
    csr_clear(CSR_MIE, MIP_MEIP | MIP_MTIP);  // 同时清除 MEIE 和 MTIE
    csr_set(CSR_MIP, MIP_STIP);
}
```

**修复 2** — `opensbi-0.9/lib/sbi/sbi_timer.c` 的 `sbi_timer_event_start()`：

```c
void sbi_timer_event_start(u64 next_event)
{
    sbi_platform_timer_event_start(sbi_platform_thishart_ptr(), next_event);
    csr_clear(CSR_MIP, MIP_STIP);
    csr_set(CSR_MIE, MIP_MEIP | MIP_MTIP);  // 重新使能 MEIE 和 MTIE
}
```

**验证**：反汇编确认生成 `csrc mie, 0x880`（清除 MEIE+MTIE）和 `csrs mie, 0x880`（设置 MEIE+MTIE）。

## 问题 2：系统在 sd_read_data_dma 处挂起

### 现象

修复问题 1 后，`kerneltrap: interrupts enabled` panic 不再出现，但系统在 `sd_read_data_dma` 处挂起（无任何输出，进程永久睡眠）。

### 调试过程

第一步，分析 DMA 等待路径。`dmac_wait_idle()` 调用 `sleep()` 等待 DMA 完成中断。如果 DMA 完成中断从未到达 S-mode，进程将永远无法被唤醒。

第二步，追踪中断转发路径。K210 上外部中断的完整转发链路：

```
DMA 完成 → PLIC 外部中断 → M-mode trap handler → 
设置 SSIP → mret → S-mode SSIP trap → devintr() → 
plic_claim() → disk_intr() → dmac_intr() → wakeup()
```

第三步，检查 PLIC M-mode 阈值。OpenSBI 的 `plic_warm_irqchip_init()` 将 M-mode PLIC 上下文阈值设为 0x7（屏蔽所有中断）。如果 xv6 不清除此阈值，M-mode 永远不会收到外部中断。

查看 `opensbi-0.9/lib/utils/irqchip/plic.c`：

```c
int plic_warm_irqchip_init(struct plic_data *plic, int m_cntx_id, int s_cntx_id)
{
    ...
    // 默认屏蔽 M-mode 阈值
    if (m_cntx_id > -1)
        plic_set_thresh(plic, m_cntx_id, 0x7);
    ...
}
```

第四步，检查 xv6 的 PLIC 初始化。`os/kernel/driver/plic.c` 的 `plicinithart()` 只设置了 M-mode 中断使能位，但从未清除阈值寄存器。

### 根因

OpenSBI 将 M-mode PLIC 阈值设为 0x7（屏蔽所有优先级的中断），xv6 从未清除此阈值。即使 M-mode 外部中断在 mie 中被使能，PLIC 也不会向 hart 发送中断信号。

### 修复

`os/kernel/driver/plic.c` 的 `plicinithart()`，K210 路径添加阈值清零：

```c
#else
  // K210: PLIC runs in M-mode, enable both UART and DISK interrupts
  uint32 *hart_m_enable = (uint32*)PLIC_MENABLE(hart);
  *hart_m_enable = *hart_m_enable | (1 << DISK_IRQ);
  uint32 *hart_m_enable_hi = hart_m_enable + 1;
  *hart_m_enable_hi = *hart_m_enable_hi | (1 << (UART_IRQ % 32));
  // set M-mode priority threshold to 0 (OpenSBI set it to 0x7).
  *(uint32*)PLIC_MPRIORITY(hart) = 0;
#endif
```

**验证**：挂起问题解决，系统不再永久等待 DMA。

## 问题 3：`scause=0x2, sepc=0x0` 指令页错误

### 现象

修复问题 2 后，系统不再挂起，但在 DMA 完成后的返回路径中崩溃：

```
scause 0x0000000000000002
sepc=0x0000000000000000 stval=0x0000000000000000
panic: kerneltrap
```

addr2line 回溯：
```
0x00000000800230bc → printf.c:205 (panic 的 printf 调用)
0x0000000080027e6e → trap.c:189 (kerneltrap 中的 myproc() 检查)
```

### 寄存器分析

| 寄存器 | 值 | 含义 |
|--------|-----|------|
| `scause` | `0x2` | 指令页错误（Instruction page fault） |
| `sepc` | `0x0` | 故障指令地址 = 0 |
| `stval` | `0x0` | 故障虚拟地址 = 0 |

CPU 尝试在地址 0 取指，该地址未映射，触发指令页错误。

### 调用链分析

故障发生在 SD 卡 DMA 读取路径，调用链为：

```
sdcard_read_sector (64字节栈帧)
  → sd_read_data_dma (内联到 sdcard_read_sector)
    → spi_init (SPI_DEVICE_0, mode_0, standard, 8-bit)
    → spi_receive_data_standard_dma (-1, DMAC_CHANNEL0, ...)
      → kalloc() — 分配 4096 字节 DMA 缓冲区
      → spi_receive_data_normal_dma (..., write_cmd=0, read_buf=kalloc页, rx_len=BSIZE=512)
        → dmac_wait_done(DMAC_CHANNEL0)
          → dmac_wait_idle
            → sleep() → sched() → swtch() — 进程睡眠等待 DMA
        → DMA 完成 → 中断 → wakeup() → swtch 恢复
        → 清理 SPI 寄存器 (ser=0, ssienr=0)
        → 返回
      → 拷贝 read_buf 数据到 rx_buff (每次取低字节)
      → kfree(write_cmd)
      → 返回
    → 返回
  → printf("sd_read_data") — 从未到达此处
```

### 已排除的可能原因

通过反汇编和代码审查，排除了以下因素：

1. **栈溢出** — 最深的调用链上栈帧总和约 800-1100 字节，远低于 4096 字节限制
2. **寄存器保存/恢复错误** — `swtch.S` 的上下文保存和 kernelvec.S 的 trapframe 保存均正确
3. **DMA 缓冲区溢出** — kalloc 分配 4096 字节，DMA 传输 2048 字节（512 项 × 32-bit），不会越界
4. **kalloc 返回 NULL** — 系统有足够内存，且 NULL 返回会导致存储错误而非指令错误
5. **DMA 配置错误** — DMA 源/目标地址、传输宽度、突发长度与参考实现一致
6. **中断转发逻辑** — 经与 RustSBI 交叉验证，MEIE/MTIE 的清除/使能时机正确

### sepc=0 的可能成因

`sepc` 变为 0 只有两种可能：

1. **`ret` 指令跳转到 0** — `jalr zero, ra, 0` 中 `ra=0`。某处栈上的 `ra` 被清零，在函数返回时加载到 `ra` 寄存器
2. **`sret` 指令跳转到 0** — trap 处理返回时 `sepc` CSR 被清零

### 调试插桩

为精确定位 ra 清零或 sepc 清零的位置，在关键路径添加了调试输出（详见下文第 4 节）。

## 中断转发机制总结

K210（v1.9.1 规范）上外部中断的完整转发流程：

```
硬件外部中断 (DMA/SPI/UART)
  ↓
PLIC 仲裁 (优先级 + 阈值过滤)
  ↓
M-mode 外部中断 (mcause=11, 无法委托给 S-mode)
  ↓
OpenSBI sbi_trap_handler:
  - 写入 stval ← IRQ_S_EXT (9)
  - 设置 mip.SSIP
  - 清除 mie.MEIE + mie.MTIE（防止嵌套 M-mode 中断）
  ↓
mret → S-mode
  ↓
S-mode 检测到 sip.SSIP + stval=9
  ↓
kernelvec → kerneltrap → devintr:
  - plic_claim() → 读取中断源
  - disk_intr() → dmac_intr() → wakeup()
  - plic_complete()
  - 清除 sip.SSIP
  - sbi_set_mie() ecall → M-mode 重新使能 MEIE+MTIE
  ↓
sret → 返回被中断的代码
```

**关键设计要点**：
1. M-mode 处理外部中断时，同时清除 MEIE 和 MTIE，防止 `mret` 前的竞态
2. S-mode 处理完成后，通过 `sbi_set_mie()` ecall 请求 M-mode 重新使能中断
3. PLIC 阈值必须在 xv6 中被清零（OpenSBI 默认设为 0x7）

## 问题 5：sepc=0, scause=2 —— 栈损坏导致的非法指令异常

### 现象

修复问题 2 后，系统在 SD 卡操作期间崩溃，出现两种变体：

**变体 A**：
```
sd_send_cmd
scause 0x0000000000000002        # 非法指令
sepc=0x0000000000000000          # 执行地址 = 0
stval=0x0000000000000000
panic: kerneltrap
backtrace: 0x800230c4 0x80027ec6 0x80027f34
```

**变体 B**（添加深层调试后）：
```
=== DEEP DEBUG FIELD ===
scause = 0x0000000000000002, stval = 0x0000000000000000
Current Hardware RA value: 0x0000000080027e40
Saved FP/RA on stack: fp=0x0000003fffffdd40, return_addr=0x0000000080027fa4
panic: kerneltrap: not from supervisor mode
backtrace: 0x800230c4 0x80027ee8 0x80027fa4
```

### 关键地址解析

| 地址 | 位置 | 说明 |
|------|------|------|
| `0x80027dee` | `kerneltrap` 起始 | C 函数入口 |
| `0x80027e40` | `kerneltrap + 0x52` | `printf("Current Hardware RA value...")` 返回地址 |
| `0x80027ef0` | `kernelvec` 起始 | S 态 trap 入口汇编 |
| `0x80027f34` | `kernelvec + 0x44` | sd 保存指令 (保存 a5, offset 112) |
| `0x80027f80` | `timervec` 起始 | M 态裸机定时器处理（不应在 OpenSBI 下执行） |
| `0x80027fa4` | `timervec + 0x24` | `li a1, 2` 指令 |
| `0x3fffffdd40` | TRAPFRAME + 0xd40 | 远超 trapframe 结构体（约 280 字节） |

### 关键发现

#### 1. fp 指向 TRAPFRAME 而非内核栈

`__builtin_frame_address(0)` 读取 s0 寄存器。在 `kernelvec.S` 中 s0 被保存/恢复：

```asm
sd s0, 56(sp)    # 保存被中断代码的 s0
...
call kerneltrap  # kerneltrap 内 s0 = 被中断代码的 s0
...
ld s0, 56(sp)    # 恢复
```

由于 trap 来自 U 态（SPP=0，变体 B），s0 包含**用户态的值**。用户进程的 s0 可指向 TRAPFRAME（该页面在用户地址空间中映射）。因此 fp=0x3fffffdd40 不一定是栈损坏——可能是用户态的正常 s0 值。

但变体 A 中 trap 来自 S 态（SPP=1），fp 却同样异常。结合 `sepc=0`（CPU 在地址 0 取指），确认是**栈损坏**导致。

#### 2. backtrace 中的 timervec 地址是垃圾数据

`0x80027fa4` 在 `timervec` 内部，但 `timervec` 是 M 态裸机代码：
```asm
timervec:
    csrrw a0, mscratch, a0    # M 态 CSR！S 态执行会触发非法指令
    ...
    csrw sip, a1              # 直接写 sip！仅 M 态可用
    mret
```

`timervec` **不应在 OpenSBI 下执行**。backtrace 中的地址是栈损坏后，backtrace 函数沿损坏的帧指针遍历，将碰巧的数据解释为返回地址的结果。

### 根因分析：嵌套 S 态中断导致栈溢出

#### OpenSBI mret 恢复 SIE 的问题

OpenSBI 的 `_trap_handler` 在 `fw_base.S` 中通过 `mret` 返回 S 态：

```asm
TRAP_RESTORE_MEPC_MSTATUS:
    REG_L   t0, SBI_TRAP_REGS_OFFSET(mepc)(sp)
    csrw    CSR_MEPC, t0
    REG_L   t0, SBI_TRAP_REGS_OFFSET(mstatus)(sp)
    csrw    CSR_MSTATUS, t0       # 恢复完整 mstatus，含 SIE 位！
```

`csrw CSR_MSTATUS` 恢复了保存时的完整 mstatus，其中 SIE 位反映的是中断发生前 S 态的中断使能状态（通常 SIE=1）。

#### 正常硬件行为

```
mret → S-mode (SIE=1), STIP/SSIP 挂起
  → 硬件立即触发 S 态 trap
  → 硬件: SPIE = SIE(1), SIE = 0, SPP = S-mode
  → kernelvec 处理 (SIE=0 ✓)
  → sret: SIE = SPIE(1) ✓
```

#### K210 v1.9.1 硬件异常

```
mret → S-mode (SIE=1), STIP/SSIP 挂起
  → 硬件触发 S 态 trap
  → 硬件: SPIE = SIE(1) ✓, 但 SIE 未清除 = 1 ✗ (BUG!)
  → kernelvec 处理 (SIE=1 ✗)
```

K210 v1.9.1 硬件在 trap 入口**正确保存 SPIE**，但**可能未清除 SIE**。

#### 嵌套中断的触发链

SIE=1 在 trap 处理期间打开，任何 M 态事件都可能触发嵌套：

```
1. 定时器中断 → M 态
2. OpenSBI: 保存 mstatus(SIE=1), 设置 STIP, mret
3. S 态: SIE=1(bug), STIP 挂起, 进入 kernelvec
4. kernelvec 保存寄存器(sp -= 256)
5. kerneltrap → clockintr → sbi_set_timer() → ecall
6. M 态 ecall → sbi_timer_event_start():
     csr_set(MIE, MEIP|MTIP)     ← M 态中断使能！
7. 若外部中断(DMAC)挂起 → 立即触发 M 态外部中断
8. OpenSBI: 设置 SSIP(stval=9), 清除 MIE, 返回 ecall
9. ecall 返回 S 态: 当前 SIE=1(bug 未清除)
10. SSIP 挂起 + SIE=1 → 嵌套 S 态 trap!
11. kernelvec 再次保存: sp -= 256 (累计 -512)
12. 嵌套处理返回(sret): SIE 可能又变回 1
13. 若再有定时器中断 → 再嵌套(sp -= 768)
14. ...反复嵌套直到内核栈(4KB)溢出
15. 栈溢出覆盖保存的寄存器 → sret 跳转到地址 0
16. sepc=0 → 非法指令异常(scause=2)
```

每次嵌套消耗 **256 字节**（kernelvec 保存 31 个寄存器 = 248 字节，对齐至 256）。内核栈 4KB，约 16 次嵌套即可溢出。

### 方案探索

#### 方案 A：sbi_trap_redirect 注入 S 态 trap（已否决）

在 OpenSBI 中使用 `sbi_trap_redirect()` 将 M 态中断注入 S 态。

**否决原因**：M 态中断可能在 SBI ecall 处理期间触发。此时 MPP=M-mode（ecall 在 M 态执行），`sbi_trap_redirect()` 检查 `prev_mode != PRV_S` 返回 `SBI_ENOTSUPP`。

```c
// sbi_trap_redirect 中的检查
prev_mode = (regs->mstatus & MSTATUS_MPP) >> MSTATUS_MPP_SHIFT;
if (prev_mode != PRV_S && prev_mode != PRV_U)
    return SBI_ENOTSUPP;  // MPP=M-mode 时在此失败
```

用户反馈："你在搞什么？"——拒绝了此方案。

#### 方案 B：OpenSBI 中清除 mstatus.SIE（已否决）

在 `sbi_trap_handler()` 中断处理分支中清除 `regs->mstatus &= ~MSTATUS_SIE`。

**否决原因**：mret 后 SIE=0，硬件 trap 入口 SPIE=SIE(0)=0。sret 恢复 SIE=SPIE=0，中断永久关闭。S 态无法得知原始 SIE 值（该信息在 M 态 mstatus 保存时已丢失），无法恢复。

### 最终方案 C：内核态通用化 SIE 清除 ✅

**修改文件**: `os/kernel/trap/trap.c:kerneltrap()`

**修改前**（仅处理 SSIP 外部中断转发）：

```c
uint64 cause_num = scause & 0xfff;
if (cause_num == 1 && r_stval() == 9) {
    // 仅当 scause=1(SSIP) 且 stval=9 时清除 SIE
    intr_off();
    sstatus = r_sstatus();
}

if((sstatus & SSTATUS_SPP) == 0)
    panic("kerneltrap: not from supervisor mode");
if((scause & 0x8000000000000000L) && cause_num != 1) {
    if(intr_get() != 0)
        panic("kerneltrap: interrupts enabled");
}
```

**修改后**（处理所有中断类型）：

```c
/*
 * K210 v1.9.1 hardware bug workaround: when OpenSBI forwards an M-mode
 * interrupt to S-mode via STIP/SSIP, mret restores the full saved mstatus
 * including SIE.  If S-mode had SIE=1, the CPU re-enters S-mode with
 * SIE=1 and the pending STIP/SSIP fires.  The hardware correctly saves
 * SIE into SPIE but may fail to then clear SIE, leaving SIE=1 inside the
 * trap handler.  Detect this and force SIE=0 to prevent nested traps that
 * would overflow the kernel stack (256 bytes per nest) and corrupt saved
 * registers (leading to sret to address 0).
 */
if ((scause & 0x8000000000000000L) && intr_get() != 0) {
    intr_off();
    sstatus = r_sstatus();
}

if((sstatus & SSTATUS_SPP) == 0)
    panic("kerneltrap: not from supervisor mode");
if((scause & 0x8000000000000000L) && (scause & 0xfff) != 1) {
    if(intr_get() != 0)
        panic("kerneltrap: interrupts enabled");
}
```

### 原理

1. 检查中断 bit（`scause & 0x8000000000000000L`）且 SIE 为 1（`intr_get() != 0`）
2. `intr_off()` 手动清除 SIE，防止当前处理期间嵌套
3. **关键**：SPIE 保持不变（硬件在 trap 入口已正确保存 SPIE = 原始 SIE）
4. sret 时自动恢复 SIE = SPIE，中断正确恢复
5. 对所有中断类型（STIP 定时器、SSIP 外部中断）均适用

### 为什么不需要改 OpenSBI

- 硬件在 trap 入口的 **SPIE 保存是正确的**（即使 SIE 清除异常）
- 内核端检测 SIE=1 并清除，保留了 SPIE 信息
- 若在 OpenSBI 中提前清除 SIE，SPIE 信息会随 SIE 一起丢失（SPIE=SIE=0）
- K210 的 bug 在于 SIE **清除**，而非 SPIE **保存**
