# 修改
`kernel/devsw/console.c` — 将控制台驱动改为非规范模式（逐字符读取）：

* 移除了内核层面的所有行编辑（退格、删除行等特殊字符处理）
* 仅回显可打印字符和换行符；控制字符（制表符 `0x09`、退格 `0x7f、C('U') 0x15` 等）不会回显，直接传递给读取者
* 此处会发生变更：每个字符都会设置 `cons.w = cons.e` 并调用 `wakeup(&cons.r)`，使 `consoleread()` 立即返回输入的内容，而非等待换行

`user/sh.c` — 逐字符输入 + 制表符补全：
* `getcmd()` 已被重写，使用 `read(0, &c, 1)` 替代 `gets()`，支持：
    * 回车`（\n）`— 确认行
    * `C('D')（0x04）`— 空行时作为 `EOF`
    * `C('U')（0x15）`— 清除当前行
    * `0x7f` 或 `\b` — 退格
    * `\t` — 制表符补全
* `complete_word()` 新增函数，用于打开当前目录、枚举目录项、将最后一个词与文件名前缀匹配。单个匹配项会直接补全并更新显示；多个匹配项会列出并重新显示提示行

# 设计过程
## 1. 问题分析：为什么 `gets()` 不行
要支持 tab 补全，shell 必须在用户按 tab 的瞬间做出反应，而不是等到按回车之后。原 shell 的 `getcmd()` 调用 `gets(buf, nbuf)`，后者调用 `read(0, &c, 1)`，但控制台驱动 consoleread() 只在收到换行符时才唤醒返回（行缓冲模式）。所以直到用户按下回车，shell 才能拿到任何数据——按 tab 时数据还在内核缓冲区里，shell 根本不知道。要拦截 tab，必须改为逐字符立即返回。

## 2. 方案选择
* 不改控制台驱动，只在 shell 层改——不可能，问题就在控制台的唤醒条件。
* 只在控制台加 tab 特殊处理——破坏分层，控制台不该知道文件系统。
* 把控制台改成完全透传，shell 管理所有显示——最干净的方案。

## 3. 控制台改动：非规范模式
删掉 `consoleintr()` 里的退格、`C('U')` 等行编辑代码，只保留 `C('P')`（进程列表，与行编辑无关）。每个字符都 `cons.w = cons.e`; `wakeup(&cons.r)`，让读取者能立即拿到。回显策略：可打印字符和 \n 由控制台回显，控制字符`（\t、0x7f、C('U') `等）不回显——shell 会处理它们的显示效果。这样 `gets()` 在其他程序里仍然能用（虽然退格字符会原样进入缓冲区，但 xv6 里几乎所有交互输入都经过 shell）。

## 4. Shell 改动：逐字符输入
新的 `getcmd()` 循环调用 `read(0, &c, 1)`，每次拿到一个字符立刻处理：

* 可打印字符 → 追加到缓冲区（控制台已经回显了，`shell` 不需要再 `echo`）
* `\b/0x7f` → 删掉缓冲区最后一个字符，用 `fprintf(2, "\b \b")` 擦除屏幕上的显示
* `\t` → 提取当前词（缓冲区最后一个空格后的部分），调 complete_word() 做补全
* `\n` → 加换行到缓冲区，返回
* `0x15 (C('U'))` → 删整行，擦除显示
* `0x04 (C('D'))` → 空行时作为 EOF

## 5. 补全逻辑
`complete_word()` 的实现很简单：
* `open(".", 0)` 打开当前目录
* 循环 `readdir(fd, &st)` 拿到所有目录项
* 跳过 `.` 和 `..`
* 用 `memcmp(st.name, prefix, plen)` 做前缀匹配
* 恰好一个匹配 → 擦掉前缀、写入补全后的名字、更新缓冲区
* 多个匹配 → 换行、列出所有匹配、重新显示提示行 + 当前缓冲区

缓冲区操作和屏幕操作需要同步：补全时先用 `\b \b` 擦掉旧前缀，再 `fprintf(2, "%s", name)` 写新名字。多个匹配不修改缓冲区，只是列出然后重绘 `$` +缓冲区内容。

## 6. 边界情况处理
* 前缀为空时不做补全（避免列出所有文件）
* 补全后不自动追加空格（用户继续打字即可）
* `\r`（K210 回车时的多余字符）在控制台层直接丢弃，shell 不会看到
* 补全后缓冲区的 `*pos` 和 `buf[*pos] = 0` 正确更新
