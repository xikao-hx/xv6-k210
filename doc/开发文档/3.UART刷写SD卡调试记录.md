# UART 刷写 SD 卡文件系统开发与调试记录

本文记录在 xv6-k210 中实现通过 UARTHS 将主机端 `target/fs.img` 写入 SD 卡的开发和调试过程。当前稳定方案使用 115200 baud 启动 shell/INFO 握手，随后主机切到 460800 baud 传输数据，板端 baud 使用实测可用的 500000。数据阶段使用分包 CRC32 校验、逐扇区 ACK/NAK。主机端会解析 FAT32，只发送实际已分配区域，而不是固定发送整个 64MiB 镜像。

## 目标

原始更新 SD 卡文件系统需要读卡器写入：

```sh
make sdcard dev-sd=/dev/sdX
```

目标是板子启动后在 shell 中运行用户程序 `burn`，主机运行：

```sh
python3 tools/burn.py /dev/ttyUSB0 target/fs.img
```

通过 UART 将 `fs.img` 写到 SD 卡起始扇区，从而更新板上 FAT32 文件系统。

## 涉及文件

- `user/burn.c`
  板端烧写程序，接收 UART 包、校验 CRC、写 SD 卡、回 ACK/NAK。

- `tools/burn.py`
  主机端发送脚本，启动板端 `burn`，发送 INFO/DATA/DONE 包，处理 ACK/NAK 和重试。

- `kernel/devsw/uartdev.c`
  raw UART 字符设备，给用户态提供二进制 UART 读写和 ioctl。

- `kernel/devsw/sdcarddev.c`
  SD 卡字符设备，提供按 512 字节扇区读写的接口，并在 raw 写盘后提供文件系统缓存失效 ioctl。

- `kernel/fs/bio.c` / `kernel/fs/fat32.c`
  FAT32 文件系统和块缓存。UART raw 写盘绕过这两层，因此刷写完成后必须失效缓存，避免连续刷写时 exec 读到旧缓存。

- `user/oled.c` / `user/oled.h`
  OLED 用户态驱动，用于烧写过程中的简要状态显示。

- `Makefile`
  生成 `fs.img`，烧写内核，读卡器写入/校验 SD 卡。

## 当前协议

主机和板端通过带 CRC32 的二进制帧通信。

帧格式：

```text
magic(4) = 55 AA 55 AA
seq(4 LE)
type(1)
plen(2 LE)
payload(plen)
crc32(4 LE)
```

CRC32 覆盖 `magic` 到 `payload` 的所有字节，使用与 Python `zlib.crc32()` 兼容的 reflected polynomial `0xEDB88320`。

包类型：

```text
PKT_INFO = 0x01  主机 -> 板端，payload 为镜像大小和板端 baud
PKT_DATA = 0x02  主机 -> 板端，payload 为 512 字节扇区
PKT_DONE = 0x03  主机 -> 板端，表示传输完成
PKT_BAUD = 0x04  主机 -> 板端，baud 切换确认/同步
PKT_ACK  = 0x81  板端 -> 主机，表示接受
PKT_NAK  = 0x82  板端 -> 主机，表示拒绝，payload 为错误码
```

ACK payload 当前包含诊断信息：

```text
reason(1)
ticks(4 LE)
```

`reason` 的含义：

```text
OK   正常写入并 ACK
DUP  收到已经写过的旧 DATA，重发 ACK，不重复写 SD
INFO INFO 阶段 ACK
DONE DONE 阶段 ACK
BAUD baud 切换阶段 ACK
```

`ticks` 在 `OK` 时表示一次 `write(sdcard_fd, ..., 512)` 花费的 `uptime()` tick 数。

## 板端流程

`user/burn.c` 的主要流程：

1. 打开 `DEV_UART` 和 `DEV_SDCARD`。
2. 初始化 OLED。
3. `UART_IOCTL_RAW_START` 进入 raw UART 模式。
4. 发送 `BURN\n` 给主机。
5. 接收 INFO，得到镜像大小并计算扇区数。
6. 检查 SD 卡容量，`SDCARD_IOCTL_SEEK` 到 0 扇区。
7. 逐扇区接收 DATA：
   - CRC 错误：回 NAK。
   - `seq < sec`：说明主机没收到旧 ACK，重发 ACK，不重复写。
   - `seq != sec` 或长度错误：回 NAK。
   - 正常：写 SD 卡，然后回 ACK。
8. 接收 DONE，回 ACK。
9. `UART_IOCTL_RAW_END` 退出 raw 模式。

关键约束：

正常数据热路径中不要刷新 OLED，也不要 `printf()`。K210 UARTHS RX FIFO 只有 8 字节，OLED I2C 刷屏和 console 输出都可能让下一包数据溢出。

## 主机端流程

`tools/burn.py` 的主要流程：

1. 打开串口，默认 115200 baud，用于 shell 启动握手。
2. 发送 `/burn\n`，等待板端 `BURN\n`。如果路径执行失败，自动尝试 `burn`、`/bin/burn`、`bin/burn`。
3. 发送 INFO 包，payload 为 `fs.img` 大小和板端 baud 设置。
4. 等待 INFO ACK。
5. 如果主机传输 baud 不是 115200，先在 115200 下发送 BAUD 请求，板端回 BAUD READY 后切到板端 baud；主机随后切到主机传输 baud，再发一次 BAUD 同步包。
6. 解析 FAT32 的 FAT 表，裁剪到最后一个已分配 cluster，只发送有效区域。
7. 按 512 字节切分待发送数据。
8. 每个扇区发送 DATA，等待 ACK/NAK。
9. ACK 的 `seq` 必须等于当前扇区，否则视为旧响应。
10. 超时或 CRC 错误时重试。
11. 最后发送 DONE，并短暂读取板端恢复 console 后打印的收尾日志。

逐包日志默认关闭，只在重试/错误时打印。需要全量详细日志时使用 `--verbose`：

```text
sec=0 attempt=1: send DATA len=512
sec=0 attempt=1: recv ACK seq=0 reason=OK ticks=0
```

```sh
python3 tools/burn.py --verbose --baud 460800 --board-baud 500000 /dev/ttyUSB0 target/fs.img
```

## raw UART 设计

`kernel/devsw/uartdev.c` 提供 raw UART 设备，底层接收缓冲由 `kernel/driver/uarths.c` 维护。

当前稳定方案使用：

- `UART_IOCTL_RAW_START`
  清空硬件 RX FIFO，切换到 raw 模式。之后 UART RX 中断不再把字节交给 console，而是写入 raw ring buffer。

- `UART_IOCTL_RAW_END`
  退出 raw 模式，清空 raw ring buffer，恢复 console 输入路径。

- `read(uart_fd, ...)`
  从 raw ring buffer 阻塞读取二进制数据。`burn.c` 会循环调用 `read()`，直到读满协议需要的字节数。

- `write(uart_fd, ...)`
  使用 `uartputc_sync()` 同步发送 `BURN\n`、ACK、NAK。

当前保留四个 ioctl：`RAW_START`、`RAW_END`、`SET_BAUD` 和 `GET_BAUD_INFO`。调试过程中用过的 `UART_IOCTL_GETC`、`UART_IOCTL_FLUSH_RX` 已经删除；板端用户程序不再逐字节 ioctl 读取 UART。`SET_BAUD` 用于 INFO ACK 之后切换数据阶段 baud，以及退出前恢复 115200；`GET_BAUD_INFO` 只用于调试显示当前分频和计算出的实际 baud。

后续补充了 `UART_IOCTL_GET_RAW_STATS`，用于在烧写结束或失败时读取 raw ring 的 dropped、buffered、capacity 和 mode。`burn` 会把 `raw_drop`、`raw_buf` 以及 CRC、协议、I/O、SD、重复包计数打印到 console；主机端 `tools/burn.py` 同步输出 elapsed、throughput、retries、timeouts、NAK 和 SD 写入 tick 汇总。这样不同 baud 的稳定性测试可以直接用日志归档，而不用临时加调试代码。

为什么改成中断 + raw ring buffer：

原始 UART RX 中断路径是：

```text
uartintr -> uartgetc -> consoleintr
```

这条路径服务 console，不适合任意二进制流。现在 `uartintr()` 在 raw 模式下会把字节写入独立 ring buffer，`burn.c` 通过 `read()` 阻塞读取该 buffer。这样不再需要逐字节 `UART_IOCTL_GETC`，也不再需要 `no_preempt` 禁止调度。

最终数据路径：

```text
normal mode:
  UART RX interrupt
    -> uartintr()
    -> uartgetc()
    -> consoleintr()

raw mode:
  UART RX interrupt
    -> uartintr()
    -> uartgetc()
    -> uart_raw_input()
    -> raw ring buffer
    -> uartdev_read()
    -> read(uart_fd, ...)
```

raw ring buffer 当前大小为 8192 字节。相比 K210 UARTHS 8 字节硬件 FIFO，它能容忍短时间调度、SD 写入和系统调用开销造成的延迟。ring 满时会丢弃新字节并记录 dropped 计数；在 230400/460800 baud、stop-and-wait 协议下没有再复现 INFO/ACK 阶段丢包。

`UART_IOCTL_RAW_START` 会先短暂关闭 RX 中断并清空硬件 FIFO，再开启 raw 模式并恢复 RX 中断，避免 console 遗留字节进入二进制协议流。`UART_IOCTL_RAW_END` 关闭 raw 模式、唤醒等待中的 reader，并清空 raw ring。

删除 `no_preempt` 后，timer interrupt 可以正常调度当前进程。UART 接收不再依赖用户态进程持续运行，而由中断负责尽快把硬件 FIFO 搬入内存。

## OLED 调试过程

最初 `burn.c` 中有多个本地 wrapper：

```c
dbg_status()
dbg_info()
dbg_line2()
dbg_hexrow()
```

后来将 OLED 驱动改成全局状态，`oled_init()` 返回成功/失败，不再向调用者暴露 I2C fd。常用接口变为：

```c
oled_write_row(row, str);
oled_printf(row, col, fmt, ...);
oled_show_hex32(row, col, value);
oled_write_hexrow(row, label, data, n);
```

同时把十六进制行显示逻辑下沉到 OLED 库中，`burn.c` 不再自己拼 16 字符缓冲。

调试中曾加入 `GOT DATA`、`SD WRITE`、`SEND ACK` 等细粒度 OLED 状态。但这些状态位于数据热路径，I2C 刷屏会干扰 UART 接收，导致 ACK 超时。因此最终稳定版只保留阶段/错误显示，并每 128 个扇区刷新一次进度。

当前进度显示：

```text
WRITING FS V2
SEC 1280
TOT 5252
PROGRESS 24%
```

这里 `SEC` 和 `TOT` 都是十进制。调试过程中曾出现：

```text
SEC 12801546
PROGRESS 16%B
```

根因是 `oled_printf()` 只覆盖新字符串，不清空整行；上一帧残留的 `1546` 或 `B` 留在行尾。修复方式是进度显示先 `snprintf()` 到 16 字符缓冲，再调用 `oled_write_row()` 整行清空并写入。

另一个显示问题是用户态 `printf/vsnprintf` 原本不支持 `%u`，导致 `SEC/TOT/PROGRESS` 格式显示异常。修复是在 `user/printf.c` 的格式化分支中增加 `%u`，因此 `printf`、`snprintf`、`oled_printf` 都可以输出无符号十进制。

## 主要问题与定位

### 1. `Timeout waiting for magic`

现象：

```text
sec=2 attempt=1 failed: timeout waiting for magic
sec=2 attempt=2 recv ACK seq=2
```

初步怀疑是 ACK 丢失或 SD 写入慢。后来通过 ACK 诊断 payload 看到：

```text
reason=OK ticks=0
```

说明 SD 写入本身不是 10 秒级卡顿。真正原因之一是 OLED 调试输出放在正常 ACK 前后，导致板端不能及时回到 UART 接收路径。

修复：

- 正常路径不再刷 OLED。
- 写 SD 后立即 ACK。
- ACK 后立即回到 `recv_msg()`。
- `tools/burn.py` 保留主机端详细诊断。

### 2. 重复 DATA 与 ACK 幂等

如果主机没收到 ACK，会重发同一个扇区。板端如果已经写入并推进 `sec`，重发包的 `seq` 会小于当前 `sec`。

修复：

```c
if (type == PKT_DATA && plen == 512 && rseq < sec) {
  send_ack_payload(uart_fd, rseq, ACK_DUP, sec);
  continue;
}
```

这样不会重复写 SD，也能让主机恢复同步。

主机端也增加 seq 校验：

```python
if seq != sec:
    # stale response
    continue
```

### 3. `Sector 1/524288` 太多

原因不是整张 SD 卡，而是 `target/fs.img` 原先固定为 256MiB：

```make
dd if=/dev/zero of=$T/fs.img bs=512k count=512
```

256MiB / 512B = 524288 sectors。

后来改为可配置：

```make
FS_SIZE_MB ?= 64
dd if=/dev/zero of=$T/fs.img bs=1M count=$(FS_SIZE_MB)
```

默认 64MiB，对应 131072 个扇区。64MiB 避免了 32MiB FAT32 簇数过低警告，同时比原来的 256MiB 少 4 倍刷写量。

后续进一步优化为：`tools/burn.py` 默认只发送 FAT32 实际使用区域。脚本读取 BPB 和 FAT 表：

- 保留 reserved sector、FSInfo、FAT1/FAT2 等元数据区。
- 扫描 FAT 表中最后一个非 0 的 cluster。
- 发送到该 cluster 结束位置。
- 免费空间尾部不再通过 UART 发送。

示例：

```text
Image: target/fs.img
Size: 2689024 bytes (5252 sectors)
FAT32 trim: full=67108864 bytes, send=2689024 bytes, saved=64419840 bytes (95%)
FAT32 layout: data_start=2050 sectors, spc=1, last_cluster=3203
```

旧行为仍可通过 `--full-image` 保留：

```sh
python3 tools/burn.py --full-image /dev/ttyUSB0 target/fs.img
```

为了验证裁剪结果而不打开串口，可以使用：

```sh
python3 tools/burn.py --dry-run --baud 460800 --board-baud 500000 /dev/ttyUSB0 target/fs.img
```

### 4. `panic: init exiting`

现象：

```text
SDHC/SDXC detected
hart 1 starting
panic: init exiting
```

原因是 `make run platform=k210` 只烧 kernel，不会写 SD 卡文件系统。虽然 make 输出中生成了 `target/fs.img`，但板子启动时读的是 SD 卡上的内容。

后续发现 `/dev/sdc` 不是块设备，而是普通文件：

```text
-rw-r--r-- 1 root root 67108864 /dev/sdc
lsblk: /dev/sdc: not a block device
```

这说明之前 `dd of=/dev/sdc` 时真实块设备不存在，`dd` 在 `/dev` 下创建了普通文件，SD 卡并没有被更新。

修复：

- `sdcard` 目标增加块设备保护：

```make
@test -b "$(dev-sd)" || (echo "$(dev-sd) is not a block device; refusing to write fs.img"; exit 1)
```

- 增加 `verify-sd` 目标，比较 SD 卡开头内容是否和 `target/fs.img` 一致。

验证方式：

```sh
make verify-sd dev-sd=/dev/sdX
```

必须使用整卡设备，如 `/dev/sdc`，不要使用分区 `/dev/sdc1`。

### 5. 高波特率切换

UARTHS 文档说明速率最高可到 5Mbps，但 K210 UARTHS 只有 8 字节 FIFO，且没有硬件流控。最终采用两阶段 baud，并把主机 baud 与板端 baud 分开：

1. 115200 下启动 `burn`。
2. INFO 中携带板端 baud 设置。
3. 板端先在 115200 下发送 INFO ACK。
4. 主机在 115200 下发送 BAUD 请求，板端回 BAUD READY。
5. 板端等待 TX FIFO 空闲后切到板端 baud。
6. 主机切到主机传输 baud，并发送 BAUD 同步包。
7. 板端收到同步包后回 BAUD ACK。
8. 数据阶段开始发送 DATA。

单独测试发现，主机设置为 230400、板端也设置为 230400 时，板端发出的 `BSET\n` 会被主机读成类似：

```text
b'\x82\xb3\x85\xb4\n'
```

这说明两端实际速率没有对齐。使用独立测试程序验证后，主机仍使用 230400，但板端请求 baud 改为约 253400，可以稳定通过：

```sh
python3 tools/uartbaud.py /dev/ttyUSB0 --baud 230400 --board-baud 253400
```

输出：

```text
Board transmitted at new baud
Sending 64-byte pattern at 230400 baud
Board reply: b'OK\n'
UART baud test passed
```

因此 `tools/burn.py` 最初默认使用主机 230400，同时把发给板端的 baud 按 1.10 倍补偿。默认等价于：

```sh
python3 tools/burn.py --baud 230400 --board-baud 253440 /dev/ttyUSB0 target/fs.img
```

如果需要复现本次手动验证的精确值，可以使用：

```sh
python3 tools/burn.py --baud 230400 --board-baud 253400 /dev/ttyUSB0 target/fs.img
```

后续继续测试 460800、921600、1500000 时，也必须先用 `tools/uartbaud.py` 找到对应的板端补偿值。由于没有硬件流控，高速下仍需严格保持 stop-and-wait，或加入更完整的窗口/重传策略。

后续实测结果：

```text
python3 tools/uartbaud.py /dev/ttyUSB0 --baud 460800 --board-baud 500000
Board transmitted at new baud
Sending 64-byte pattern at 460800 baud
Board reply: b'OK\n'
UART baud test passed
```

而 576000、768000、921600 多组补偿值都出现 `BSET` 或返回包无法干净解码的问题。当前工程的 `make download` 使用稳定组合：

```sh
python3 tools/burn.py --baud 460800 --board-baud 500000 /dev/ttyUSB0 target/fs.img
```

64MiB 完整镜像在 460800 baud 下理论仍需二十多分钟；但配合 FAT32 裁剪后，当前实际发送约 2.6MiB，刷写时间大幅缩短。

### 6. `/burn`、`/bin/burn` 与 fs.img 未更新

`scripts/mkfs.py` 会把 `build/user/_*` 用户程序同时写入根目录和 `/bin`：

```python
vfs.open("/" + name, "wb").write(data)
vfs.open("/bin/" + name, "wb").write(data)
```

因此理论上 `_burn` 会生成：

```text
/burn
/bin/burn
```

调试过程中出现：

```text
exec /burn failed
exec burn failed
/bin/burn 成功
```

以及后续全部路径失败：

```text
Board shell could not exec burn program
```

定位时发现一个容易误判的问题：`build/user/_burn` 已经是新版，但 `target/fs.img` 仍是旧时间戳。原因是 Makefile 还包含 `$(UBUILD)/_uartbaud`，而 `user/uartbaud.c` 已经被删除，`make fs` 在生成镜像前失败，导致用户以为刷了新版，实际仍在反复刷旧 `fs.img`。

修复：

- 从 `UPROGS` 移除已经不存在的 `_uartbaud`。
- 重新执行 `make fs`。
- 用 `stat` 确认 `target/fs.img` 时间戳晚于 `build/user/_burn`。

验证：

```text
build/user/_burn  11:11:32
target/fs.img     11:11:37
```

`tools/burn.py` 也增加了 `--burn-cmd` 和 fallback 顺序。默认先尝试 `/burn`，失败后尝试 `burn`、`/bin/burn`、`bin/burn`。如果当前板子只有 `/bin/burn` 可执行，可以直接指定：

```sh
python3 tools/burn.py --burn-cmd /bin/burn --baud 460800 --board-baud 500000 /dev/ttyUSB0 target/fs.img
```

### 7. 连续刷写第二次 `usertrap(): addr is illegal`

现象：

第一次刷写成功：

```text
Sector 5252/5252 (100%)
Transfer completed successfully! ACK reason=DONE ticks=0
```

不重启马上第二次执行：

```text
Sending '/burn' command to board at 115200 baud...
Waiting for BURN signal...
Timeout waiting for BURN, retrying '/burn' (1); last serial='/burn\nusertrap(): addr is illegal\n$ '
```

原因是板端 `/burn` 通过 `sdcarddev_write()` 直接调用 `sdcard_write_sector()` 覆盖 SD 卡，绕过了 FAT32 文件系统层和 buffer cache。第一次刷写成功后，磁盘内容已经换成新版 `fs.img`，但旧 kernel 内存中仍缓存着旧的 FAT、目录项或文件块。第二次 shell `exec /burn` 时，`exec()` 可能读到“旧缓存 + 新磁盘”的混合内容，导致 ELF 装载错误，用户程序一启动就触发非法地址异常。

修复内容：

- `kernel/fs/bio.c`
  增加 `binvalidate(dev)`，将指定设备中 `refcnt == 0` 的 buffer 标记为 invalid。

- `kernel/fs/fat32.c`
  增加 `fat32_invalidate()`，清除空闲目录项缓存，避免路径查找命中旧目录项。

- `kernel/devsw/sdcarddev.c`
  增加 `SDCARD_IOCTL_INVALIDATE_CACHE`，统一调用 `fat32_invalidate()` 和 `binvalidate(0)`。

- `user/burn.c`
  在 DONE 成功后、退出 raw UART 前调用：

```c
int cache_rc = ioctl(sdcard_fd, SDCARD_IOCTL_INVALIDATE_CACHE, 0);
```

为了验证该修复是否真的在板端 kernel 生效，`burn.c` 成功结束时会打印：

```text
burn: done sectors=5252 bytes=2689024 cache=0
```

其中 `cache=0` 表示新版 kernel 已经执行缓存失效；如果看到 `cache=-1`，说明板子仍是旧 kernel，不认识该 ioctl。

注意：这是 kernel 改动。只刷 `fs.img` 不会让它生效，必须先：

```sh
make run platform=k210
```

然后再刷 SD：

```sh
python3 tools/burn.py --baud 460800 --board-baud 500000 /dev/ttyUSB0 target/fs.img
```

最终验证输出：

```text
Sector 5252/5252 (100%)
Transfer completed successfully! ACK reason=DONE ticks=0
Board console tail:
burn: done sectors=5252 bytes=2689024 cache=0\n$
```

看到 `cache=0` 后，连续第二次刷写不应再因为旧 FAT/buffer cache 触发 `usertrap(): addr is illegal`。

## SD 卡内容验证

读卡器写入：

```sh
make sdcard dev-sd=/dev/sdX
```

验证 SD 卡内容：

```sh
make verify-sd dev-sd=/dev/sdX
```

该目标会：

1. 检查 `dev-sd` 是否能被 `lsblk` 识别。
2. 读取 sector 0 offset 82 的 FAT32 签名。
3. 用 `cmp -n` 比较 `target/fs.img` 和 SD 卡起始区域。

如果完全一致，会输出：

```text
SD content matches target/fs.img
```

## 当前稳定用法

生成 64MiB 文件系统：

```sh
make fs
```

第一次或 SD 卡损坏时，用读卡器写入：

```sh
make sdcard dev-sd=/dev/sdX
make verify-sd dev-sd=/dev/sdX
```

板子能进入 shell 后，可以通过 UART 更新：

```sh
python3 tools/burn.py --baud 460800 --board-baud 500000 /dev/ttyUSB0 target/fs.img
```

如果当前板子根目录 `/burn` 不可执行，但 `/bin/burn` 可执行，可以指定：

```sh
python3 tools/burn.py --burn-cmd /bin/burn --baud 460800 --board-baud 500000 /dev/ttyUSB0 target/fs.img
```

详细日志：

```sh
python3 tools/burn.py --verbose --baud 460800 --board-baud 500000 /dev/ttyUSB0 target/fs.img
```

只检查将要发送多少数据，不打开串口：

```sh
python3 tools/burn.py --dry-run --baud 460800 --board-baud 500000 /dev/ttyUSB0 target/fs.img
```

刷写成功后，如果看到：

```text
Board console tail:
burn: done sectors=... bytes=... cache=0
```

说明新版 kernel 已完成 FAT/buffer cache 失效，可以继续不重启刷下一次。

`make run platform=k210` 只烧 kernel，不写 SD 文件系统。更新了 kernel 之后仍需要再执行 UART burn 或读卡器写 SD。Makefile 现在会打印提示：

```text
K210 kernel flashed. SD filesystem is not updated by make run.
Use 'make sdcard dev-sd=/dev/XXX' or 'python3 tools/burn.py /dev/ttyUSB0 target/fs.img' to update fs.img.
```

## 后续优化方向

### 1. 多扇区窗口

当前是 stop-and-wait：每个 512 字节 DATA 等一个 ACK。吞吐受 RTT 和 SD 写入影响。后续可以考虑一次发送多个扇区，再统一 ACK bitmap，但实现复杂度会明显增加。

### 2. SD 多块写入

当前每扇区调用一次 `sdcard_write_sector()`。可考虑 SD 多块写命令提升写入效率，但需要改 SD 驱动，并处理失败回滚/重试策略。

### 3. 构建系统与镜像内容一致性

当前 `scripts/mkfs.py` 会扫描 `build/user/_*`，因此旧的构建产物即使不在 `UPROGS` 中，也可能被写入 `fs.img`。调试中就出现过 `user/uartbaud.c` 已删除，但 `build/user/_uartbaud` 仍被打进镜像的情况。

后续可以把 `mkfs.py` 改为读取 Makefile 导出的用户程序清单，或在 `make fs` 前清理不在 `UPROGS` 的旧 `_` 程序，避免镜像内容和 Makefile 预期不一致。

    ├─ 读取 FAT32 BPB
    ├─ 扫描 FAT 表

                                      ├─ 检查 SD 容量
