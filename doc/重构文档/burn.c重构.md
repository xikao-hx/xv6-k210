# burn.c — 代码审查与重构建议

审查日期：2026-06-15  
审查文件：`user/app/burn.c`  
运行平台：K210（RISC-V，几 MB RAM，无 MMU）

---

## 总评

功能上**能跑**，协议设计（魔数同步 + 序号去重 + CRC32 校验 + 可选的波特率切换）合理，容错机制完整。但从代码质量角度看，存在**变量遮蔽、重复代码、可读性差**等明显问题，属于"能工作但粗糙"的阶段。

**评分：6/10**

---

## P0 — 必须修复

### 1. 局部变量遮蔽导致栈浪费 + 维护隐患

`main()` 第 379 行声明了 `payload[512]` 和 `seq` / `plen`，然后在：

- 第 455 行（内层 while）**重新声明**了 `uint8 payload[512]` 和 `uint16 plen`
- 第 527 行（WAIT DONE 块）**再次声明**了 `seq`、`payload[512]`、`plen`

外层 `payload` 在 INFO 阶段后几乎不再使用，但编译器仍为其分配了栈空间。内层重新声明导致：

- 栈上额外多占 **1.5 KB**（K210 总 RAM ~8 MB，但内核栈有限）
- 阅读代码的人会困惑：到底用的是哪个 `payload`？

**修复方案**：删除内层所有重复声明，统一使用 main 栈帧顶部的 `payload`/`seq`/`plen`。

### 2. 错误/正常退出路径的重复代码

`fail` 标签（第 556-564 行）和正常出口（第 539-553 行）都有这样一段：

```c
if (transfer_baud != CONSOLE_BAUD)
    ioctl(uart_fd, UART_IOCTL_SET_BAUD, CONSOLE_BAUD);
get_uart_raw_stats(uart_fd, &raw_stats);
ioctl(uart_fd, UART_IOCTL_RAW_END, 0);
```

除了 printf 内容不同，其余完全重复。

**修复方案**：提取为 `cleanup()` 函数，或用 `goto finish` 统一出口，`fail` 和正常路径分别设置错误状态后跳到同一段 cleanup。

## P1 — 强烈建议修改

### 3. CRC32 运行时表初始化

```c
static uint32 crc32_tab[256];
static int    crc32_inited = 0;
```

每次 `crc32()` 调用都要检查 `crc32_inited`。虽然开销不大，但在烧录数千扇区时反复检查同一条件。

**修复方案**：

- 方案 A：在 `main()` 开头显式调用 `crc32_init()`，去掉运行时检查
- 方案 B：用 `__attribute__((constructor))` 自动初始化
- 方案 C：编译时预生成查表（用 Python 脚本生成 .c 数组）

推荐方案 A，最简单且消除所有运行时开销。

### 4. OLED 进度步进策略过于粗糙

`OLED_PROGRESS_STEP = 128`，意味着：

| 总扇区数 | 进度刷新次数 |
|----------|------------|
| 256（~128 KB） | 2 次（0%, 100%） |
| 1024（~512 KB） | 7 次 |
| 8192（~4 MB） | 64 次 |

小镜像几乎没有进度反馈，大镜像中间等待时间太长才刷一次。

**修复方案**：改用动态步进

```c
// 至少刷新 20 次，但每两次之间不少于 1 个扇区
uint32 step = MAX(1, nsectors / 20);
// 也可基于时间：每 500ms（由 uptime 测量）刷新一次
```

或基于时间刷新（在 `sec++` 处检查 `uptime()` 差值）。

### 5. `send_msg` / `recv_msg` CRC 计算代码不对称

发送时一次 CRC 算 11 字节 `hdr`（magic + seq + type + plen）：

```c
uint32 csum = crc32(hdr, 11, 0);
```

接收时拆成 magic + hdr：

```c
uint8 magic[4] = {0x55, 0xAA, 0x55, 0xAA};
uint32 csum = crc32(magic, 4, 0);
csum = crc32(hdr, 7, csum);
```

两段代码数学上等价，但**风格不统一**。阅读者需要停下来推导才知道是等价的。

**修复方案**：发送方也拆成两步，与接收方对称：

```c
uint8 magic[4] = {0x55, 0xAA, 0x55, 0xAA};
uint8 hdr[7];  // seq(4) + type(1) + plen(2)
// ... 填充 hdr ...
uint32 csum = crc32(magic, 4, 0);
csum = crc32(hdr, 7, csum);
// payload 部分 ...
```

## P2 — 建议优化

### 6. `recv_msg` 的重新同步种子逻辑可读性差

第 230-236 行，假魔数检测失败后重新同步的种子回填：

```c
sync[0] = hdr[5];
sync[1] = hdr[4];
sync[2] = hdr[3];
sync[3] = hdr[2];
si = 2;
```

功能正确但难以快速理解。倒序赋值 + `si = 2` 的组合要画图才能想明白。

**修复方案**：改用环形缓冲区手工推入，或加 ASCII 示意图注释：

```c
// 将读到的 7 字节 header 中紧跟在假魔数后面的 4 个字节
// (hdr[0..3]) 喂给滑动窗口，避免重叠的真魔数被跳过。
// 窗口内容变为: [hdr[3], hdr[2], hdr[1], hdr[0]]
// si 指向 hdr[0] 的位置，即 si = 3（最后写入位）
```

### 7. 缺少看门狗喂食

烧录循环是纯轮询（第 449 行 `for` 循环），如果扇区数量很大（例如 512 MB SD 卡对应 ~1M 扇区），程序在该循环中停留时间极长。如果系统启用了硬件看门狗，会触发复位。

**修复方案**：在 `sec++` 后（第 516 行）周期性喂狗：

```c
if (sec % 256 == 0)
    watchdog_feed();  // 假设硬件支持
```

即使当前未用看门狗，这也是一个文档化的防御性编程。

### 8. `send_ack_payload` 在 `ACK_INFO` 时浪费 4 字节

```c
send_ack_payload(uart_fd, seq, ACK_INFO, 0);
```

`ACK_INFO` 的 ticks 字段主机端可能忽略，但仍发送了 4 字节的 0。单次影响很小，但不干净。

**修复方案**：可以不改（影响太小），或者为 `ACK_INFO` 单独定义无负载的确认包类型。

### 9. 逐字节读取效率问题

`read_bytes()` 内部逐次调用 `read(fd, buf+off, n-off)`，在中断驱动的 UART 下每次调用至少有一次上下文切换。115200 波特率下 512 字节约需 44 ms 传输，每次 `read` 可能只拿到 1 字节。

**修复方案**：如果底层驱动支持，改用较大的缓冲区批量读取。或者至少在内层加一次 `read` 尽可能多地读取剩余数据。

## 推荐重构方案（按顺序执行）

1. **修复 P0**：删除内层变量遮蔽，提取清理函数 → 消除栈浪费和重复代码
2. **优化 CRC**：`main()` 开头一次初始化，去掉运行时检查
3. **动态 OLED 步进**：基于时间或动态计算粒度刷新进度
4. **对称 CRC 风格**：发送方也拆 magic + header
5. **改进 `recv_msg` 可读性**：重写同步种子回填，加注释
6. **加看门狗喂食**（如果硬件支持）
7. **最终**：全面审视代码注释风格、命名一致性

---

## 与主机端协议的对等影响

如果改动 `send_msg` 的 CRC 计算范围或报文格式，需同步修改主机端 `tools/burn.py`。建议先确认主机端已经稳定，再考虑协议级改动。非协议级改动（变量遮蔽、重复代码、步进策略）不影响主机端，可直接修复。
